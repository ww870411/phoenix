  #### 1️⃣ Bug 1：库存计算缺少“非负截断”（Negative Inventory 导致净缺口虚增）

  • 代码定位： backend/projects/insulation_pipe_supply_2026/api/workspace.py  第 500-502 行：
    station_inventory_qty = total_arrived_qty - total_usage_qty
    inbound_pipeline_qty = pending_arrival_qty
    net_gap_qty = max(plan_total_qty - inbound_pipeline_qty - station_inventory_qty, 0)

  • 逻辑缺陷：
  现场的“实际使用量（ total_usage_qty ）”是由现场负责人每天按自然日填报并保存入库的，但“确认到货（ total_arrived_qty ）”由于负责人未及时在系统上点击“确认到货”按钮，在实际中必然存在录入滞后性。
  这会导致账面上  total_usage_qty > total_arrived_qty ，库存  station_inventory_qty  变为负数（例如  -50  根）。
  当其代入净缺口计算公式： max(plan_total_qty - inbound_pipeline_qty - (-50), 0)  时，由于减去负数相当于加上正数，“三日净缺口（ net_gap_qty ）”会被凭空虚增 50                                                         
  根！这在现实中会向管厂传达错误的缺口虚假情报，诱导管厂超额发货，打乱供应链节奏。
  • 修复建议：库存计算必须增加非负限制： station_inventory_qty = max(total_arrived_qty - total_usage_qty, 0) 。

#### 3️⃣ Bug 3：缺少到货和使用量日期上限切断，导致“历史总库存在未来时间段被污染”
  • 代码定位： supply_management_service.py  中的  list_usage_totals()  和  list_arrival_aggregates() ，对历史到货及使用数据进行无差别汇总。
  • 逻辑缺陷：
  这几个 SQL 都是无视日期范围、对 全部历史数据 进行求和。但这与拍板规则 14 发生了冲突：
  │
  │     14. 展示层实际使用、库存、累计量等默认只推进到  show_date - 1 
  │     如果库管员或现场负责人在  show_date  之后有实际使用量录入，或者有新的到货确认发生，系统算出来的  station_inventory_qty  和  net_gap_qty 
  │     会把这批“未来的到货和使用量”直接算入。这导致展示层实际展示的“累计使用量（ total_usage_qty ）”如果只展示到  show_date - 1 ，但背后用于计算库存和缺口的  station_inventory_qty  和  net_gap_qty 
  │     却用的是包含未来数据的数值，造成 UI 上明显的账实冲突！

  • 修复建议：在进行库存与缺口计算时， SUM(usage_qty)  与  SUM(arrived_qty)  必须加入  usage_date < show_date  的日期过滤条件。
  • 修改情况（2026-05-24）：已修复。供给侧库存与缺口计算使用的到货量、使用量汇总已增加 `show_date` 截断，展示层累计量默认只推进到 `show_date - 1`。

#### 6️⃣ Bug 6：施工接收确认与库管确认缺少“数量上限”强约束校验

  • 代码定位： supply_management_service.py  第 578-579 行：
    if normalized_received_qty > arrived_qty:
        raise HTTPException(status_code=422, detail="施工接收数量不能大于到货数量")
  在第 598-644 行的库管手续确认函数  update_delivery_warehouse_record  中，完全没有任何数量相关的校验或赋值，只改写了状态为  completed ！
  • 逻辑缺陷：
      • 在施工接收确认中，代码只限制了  received_qty <= arrived_qty 。如果运过来的保温管发生了漏油、内管变形等物理损耗，施工单位调减了接收数量时（例如到货 10 根，接收 8 根，损耗 2 根），
      • 那么，这发生损耗的 2 根管子，在系统算库存  总到货量 - 总使用量  时：
      由于  total_arrived_qty  是按  COALESCE(arrived_qty, received_qty, shipped_qty)  汇总的（它依然取  arrived_qty  也就是 10 根），
      这导致那坏掉的 2 根管子永远留在现场库存中充当“可用于施工的健康库存”去抵扣未来的计划，导致系统算出来的净缺口永远偏小，面临现场断料风险！
  • 修复建议：一旦确认了  received_qty  小于  arrived_qty ，库存计算公式中累加的“到货量”必须以施工单位实际接收的  received_qty  为准，或者将差值（损耗量）自动记录到  tube_inventory_adjustment                        
  （库存调整表）中进行扣减。
  • 修改情况（2026-05-24）：已修复到当前一期口径。到货量汇总已改为优先采用 `received_qty`，当施工接收数量小于到货数量时，库存与缺口计算不再继续把差额计为可用库存。

  #### 7️⃣ Bug 7：在途耗时计算 Aware 与 Naive Datetime 直接相减导致 Python 运行时崩溃
  • 代码定位： supply_management_service.py  中的  format_delivery_elapsed ：
    total_seconds = max(int((current - shipped_at).total_seconds()), 0)
  • 逻辑缺陷：
   shipped_at  是从前端接收并带时区的 aware datetime，而  arrived_confirm_at  或是数据库  NOW()  在 SQLAlchemy 的 naive mapping 下返回的是 naive datetime（无时区）。
  直接将 naive datetime 与 aware datetime 相减，在 Python 运行时会 100% 抛出  TypeError: can't subtract offset-naive and offset-aware datetimes  异常！
  这将导致发货单查询、物流记录展示等核心页面只要存在到货确认时间，访问就会瞬间报  500 Server Error  彻底崩溃！
  • 修复建议：在进行减法运算前，必须通过  shipped_at.replace(tzinfo=None)  或统一  .astimezone()  保证两边类型完全一致。
#### 8️⃣ Bug 8：异常挂起态发货单导致库存暴跌与虚假缺口报警
  • 代码定位： supply_management_service.py  中  list_arrival_aggregates() ：
    SUM(
        CASE
            WHEN status IN ('pending_receive', 'pending_warehouse', 'completed')
                THEN COALESCE(arrived_qty, received_qty, shipped_qty)
            ELSE 0
        END
    ) AS total_arrived_qty

  • 逻辑缺陷：
  在计算  total_arrived_qty  时，仅提取了  pending_receive 、 pending_warehouse 、 completed  三种状态。
  如果在现场业务中，发生了异常挂起（例如设置了  abnormal_flag = true  或自定义了状态），只要它的状态不属于这三个，它原本已经确认到货的数量在系统中会瞬间被归 0！
  这会导致当前库存  total_arrived_qty - total_usage_qty  瞬间暴跌，净缺口陡然虚增，触发管厂端的大面积供货误报。
  • 修复建议：只要发货单的  arrived_qty  已经大于 0 且发货单未被撤销（ status != 'cancelled' ），无论其当前处于何种中间处理状态（包括异常状态），其到货量都应当被计入库存。
  • 修改情况（2026-05-24）：已修复到当前一期口径。到货量汇总条件已从固定状态集合改为“非 `cancelled` 且已确认到货”，避免异常挂起态把已到货数量错误清零。
