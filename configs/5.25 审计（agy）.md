## 二、核心致命与高危漏洞专题深度起底 (Top 10)

本节针对本次审计中发现的 10 个最致命、高危的安全与逻辑缺陷进行深度剖析。

### 🚨 1. [数据库/后端] BUG #1 — `order_no` 唯一索引与初始空值冲突，高并发创建发货记录必直接报错瘫痪 (CRITICAL)
- **缺陷位置：** `tube_schema_init.sql` L123-124 与 `supply_management_service.py` L250-329
- **逻辑错误：** 
  建表 SQL 设定了 `order_no` 为全局唯一索引：
  ```sql
  CREATE UNIQUE INDEX IF NOT EXISTS uq_tube_delivery_order_no ON tube.tube_delivery (order_no);
  ```
  但在后端发货单创建代码中，采用的是“先 INSERT 插入空单号，再 UPDATE 回写正式单号”的两步操作：
  ```python
  # create_delivery_record() L289 中初始插入：
  # order_no = :order_no  (此时传入的值是空字符串 "")
  ```
- **业务危害：** 
  PostgreSQL 唯一索引会把空字符串 `''` 视为相同值（与 `NULL` 不同，`NULL` 不会触发冲突）。当有两个供应商或同一供应商同时发出两笔发货单时，在极短的并发间隙中，两笔记录的 `order_no` 都是 `''`，**第二个插入请求会直接被数据库唯一性约束拦截报错，导致发货记录创建失败！** 这是一个致命的系统可用性阻断 bug。
- **修复方案：** 
  将数据库唯一索引改写为偏索引（Partial Index），排除空值或空字符串：
  ```sql
  CREATE UNIQUE INDEX IF NOT EXISTS uq_tube_delivery_order_no 
  ON tube.tube_delivery (order_no) WHERE order_no IS NOT NULL AND order_no <> '';
  ```

### 🚨 2. [后端服务] BUG #2 — 配置中完全缺失 `username` 字段，导致非管理员的普通角色登录后看到的站点/主体列表恒为空 (CRITICAL)
- **缺陷位置：** `config_service.py` L121-161 与 `tube_config.json`
- **逻辑错误：** 
  系统在解析用户所属权限及能管理的换热站时，提取了 `manager_assignments` 及 `construction_units` 中的字段进行比对：
  ```python
  candidate_keys = {
      str(item.get("manager_id") or "").strip(),
      str(item.get("manager_name") or "").strip(),
      str(item.get("username") or "").strip(),  # ← 尝试读取 username
  }
  ```
  但翻阅 `tube_config.json` 后发现，**配置实体中根本就没有 `username` 字段！**
- **业务危害：** 
  若现场负责人的登录账号（如 `zhang_san`）不同于配置里的 `manager_id`（如 `site_manager_a`）或中文字名，则**永远无法匹配权限**。这导致所有登录的普通角色（负责人、施工方、管厂）进入系统后其数据权限集恒为空，导致界面全部变成一片空白，业务完全无法开展。
- **修复方案：** 
  在 `tube_config.json` 中的各用户关联对象上补齐 `username` 键值，或者让 `username` 直接与权限账号表同步。

### 🚨 3. [后端 API] BUG #3 — `show_date` 混用导致供给侧看到的三日滚动计划日期全部算错 (CRITICAL)
- **缺陷位置：** `workspace.py` 第 663-664 行
- **逻辑错误：** 
  在供给侧汇总接口 `get_supply_management_demand_summary` 中，统计未来三日计划的计划窗口推算，错误地使用了展示口径 `show_date`，而不是采集口径 `plan_start_date`：
  ```python
  show_date = get_configured_show_date(payload)
  plan_dates = build_plan_dates(show_date)  # ← 致命错误：应使用 plan_start_date
  ```
- **业务危害：** 
  `show_date` 通常会被管理员推进（通常比 `plan_start_date` 提前 1-2 天）。由于该混用错误，`list_plan_totals` 会在统计未来三日计划量时，多算了过去几天（已无填报数据）的计划，同时彻底**漏掉了未来最关键的填报计划日期**。这导致供给侧看到的“三日滚动计划量”和“三日净缺口”全部错乱，直接破坏了管厂的发货决策基准，面临现场断料断工风险。
- **修复方案：** 
  修正为使用 `plan_start_date`：
  ```python
  plan_start_date = get_configured_plan_start_date(payload)
  plan_dates = build_plan_dates(plan_start_date)
  ```

### 🚨 4. [前端] BUG #4 — `normalizePendingRows` 经典 `||` Falsy 覆盖 Bug，导致手动确认到货 `0` 根时被篡改为全额到货 (CRITICAL)
- **缺陷位置：** `DemandManagementView.vue` L549-557
- **逻辑错误：** 
  在对接口返回的发货单进行前台本地初始化时，采用了 `||` 做兜底值处理：
  ```js
  arrivedQty: Number(row.arrived_qty || row.arrivedQty || row.shipped_qty || row.shippedQty || 0),
  arrivalConfirmQty: Number(row.arrived_qty || row.arrivedQty || row.shipped_qty || row.shippedQty || 0),
  ```
- **业务危害：** 
  当发货单在运送中发生重大意外（如车祸损坏，或者整车装错型号），现场负责人确认**到货数量为 `0`** 时。由于 `0` 在 JavaScript 中属于 **falsy 值**，`row.arrived_qty || row.shipped_qty` 逻辑会直接越过 0，而**强行采用 shipped_qty（发货时装车数量，如 50 根）**作为到货初始值！这导致数据直接失真，不仅屏蔽了到货为 0 的异常，还自动产生了虚假到货的坏账。
- **修复方案：** 
  将所有的 `||` 升级为 Vue3/ES6 推荐的 `??`（空值合并运算符）：
  ```js
  arrivalConfirmQty: Number(row.arrived_qty ?? row.arrivedQty ?? row.shipped_qty ?? row.shippedQty ?? 0),
  ```

### 🚨 5. [前端] BUG #5 — 库管多选台账已到货/已完成的送货单“在途时长”在确认后仍像秒表般无限增长 (HIGH)
- **缺陷位置：** `WarehouseManagementView.vue` 第 413-434 行
- **逻辑错误：** 
  在 Warehouse 页面中，`formatDeliveryElapsedDisplay` 提取的是接口原始数据（snake_case 字段 `row.shipped_at`），其时间统计终止条件代码为：
  ```js
  const endValue = row.arrived_confirm_at ? new Date(row.arrived_confirm_at) : new Date(nowTick.value)
  ```
  但经核对，Warehouse 页面在获取数据进行转换时，**根本就没有把后端返回的 `arrived_confirm_at` 字段存入行对象**，或者命名为了 camelCase。
- **业务危害：** 
  对所有“已到货待接收”甚至“已完成库管确认”的历史发货单，`row.arrived_confirm_at` 判定恒为 `undefined`。这导致 `endValue` 始终 fallback 采用当前本地时间的 `nowTick`。**已经到货很久的历史送货单，其“在途时间”在网页上每过一分钟依然在疯狂累加**，数据完全失真，无法作为运送时效统计参考。
- **修复方案：** 
  确保 `arrived_confirm_at` 字段命名与前后端转换大小写完全统一：
  ```js
  const endValue = row.arrivedConfirmAt ? new Date(row.arrivedConfirmAt) : new Date(nowTick.value)
  ```

### 🚨 9. [前端/后端] BUG #9 — JSON 配置文件与填报状态文件多用户并发写回导致丢失数据漏洞 (HIGH)
- **缺陷位置：** `workspace.py` L1113-1169 与 `config_service.py` L22-42
- **逻辑错误：** 
  系统将计划填报提交状态和全局配置存放在 `.json` 文件中。然而，在执行 `load` → `modify` → `save` 的整个 read-modify-write 周期中，**没有加任何线程锁或文件锁**：
  ```python
  current_status = load_station_submission_status()
  # 内存中追加数据...
  save_station_submission_status(current_status)
  ```
- **业务危害：** 
  若换热站A和换热站B的现场负责人在 1 秒内同时点击“提交填报状态”，由于时差，先提交者的修改可能会在后提交者写盘时被**全量覆盖并抹除**，导致严重的填报状态漏登与数据丢失。
- **修复方案：** 
  使用 `filelock` 库或线程排他锁对 JSON 读写事务进行锁保护。

### 🚨 10. [前端] BUG #10 — `selectedSupplyEntityId` 三个 Watcher 循环依赖，导致 API 重复调用和状态死锁 (HIGH)
- **缺陷位置：** `SupplyManagementView.vue` 第 925-977 行
- **逻辑错误：** 
  供给侧页面中，多角色状态切换衍生了 3 个相互引用的 watcher：
  - `selectedSupplyEntityId` 变更时会把值回写给 `deliveryForm.supplyEntityId`；
  - `deliveryForm.supplyEntityId` 变更时若为全局管理员又会将值回写给 `selectedSupplyEntityId`；
  - `selectedSupplyEntityId` 还有第二个 watcher，只要变化就立刻重新触发 `loadDeliveries()`。
- **业务危害：** 
  当管理员在下拉框中切换管厂时，这三个侦听器会在响应式回路中互相循环调用，极易触发 Vue 内部防溢出的死循环保护，并使得 `loadDeliveries()` API 在一瞬间被**重复调用多次**，页面加载出现卡顿和网络请求浪费。
- **修复方案：** 
  合并冗余 watcher，避免表单模型和页面当前全局状态的双向交叉侦听，改用单向流或显式事件处理。

## 三、其他重要审计缺陷清单 (去重汇总)

本节将前后端审计中发现的 70 个中危 (MEDIUM) 与低危 (LOW) 问题中，提炼出 10 个直接影响核心业务正确性的缺陷：

### 1. 负库存穿透引发缺口虚增 (MEDIUM | 后端)
- **缺陷位置：** `workspace.py` L690
- **隐患：** 
  库存计算公式为 `station_inventory_qty = total_arrived_qty - total_usage_qty`。在数据录入滞后时，库存极易算为负数。由于没有 `max(..., 0)` 非负保护，负库存会在净缺口公式 `- station_inventory_qty`（减去负数相当于加上绝对值）中**反向虚增缺口量**，误导供应商盲目生产发货，导致库存堆积。

### 2. 施工多选台账批量确认部分失败导致状态不一致 (HIGH | 前端)
- **缺陷位置：** `WarehouseManagementView.vue` L595-613
- **隐患：** 
  库管人员可以多选送货单批量完成确认。然而，前端代码采用了 `for (const row of selected) { await confirm(...) }` 的串行循环调用方式。如果中途发生网络抖动，第 3 条失败了，前 2 条已成功。前端只抛出失败错误，用户根本无法得知究竟哪些成功、哪些失败，成功提示里仍会错误地显示处理了全部 5 条。

### 3. 已撤销的发货单号仍可被“继续发货”(MEDIUM | 前端)
- **缺陷位置：** `SupplyManagementView.vue` L335-353
- **隐患：** 
  在供给侧记录中，当 status 为 `cancelled` 时，由于行数据依然包含 `shipmentNo`，"继续该车次"按钮依然显示并可被点击。这允许供应商用已经作废的车次号去创建新的发货草稿，导致车次数据和异常标记发生污染。

### 4. 凌晨跨天日期时区偏离 1 天 (MEDIUM | 前端)
- **缺陷位置：** `DemandManagementView.vue` L472
- **隐患：** 
  `getTodayString()` 使用 `new Date().toISOString().slice(0, 10)` 提取日期。在东八区（中国）环境下，每天凌晨 0:00 - 8:00 之间调用时，`toISOString()` 返回的 UTC 日期会**比中国本地日期少一天**，导致这段时间内登录的用户填报计划或拉取数据直接错位。

### 5. 计划可编辑判断脆弱，变更日期范围即失效 (HIGH | 前端)
- **缺陷位置：** `DemandManagementView.vue` L456
- **隐患：** 
  `isPlanDateEditable(index)` 采用 `index >= planDates.value.length - editableDays` 判断。该逻辑严重依赖后端正好只返回 3 天数据。如果未来后端为了实现趋势对比而返回 5 天或 7 天数据，此逻辑会让最后 N 天皆可编辑，从而把本该锁定的历史日期重新开放出来。

### 6. 物流台账遗失已完成历史记录 (MEDIUM | 后端)
- **缺陷位置：** `workspace.py` L1219
- **隐患：** 
  需求侧物流确认记录接口中硬编码 `and row.get("status") in {"pending_arrival", "pending_receive", "pending_warehouse"}`。这导致用户一旦完成了库管确认，该物流记录便在列表中人间蒸发，现场负责人无法回查任何历史已完成的物流单据。

### 7. SQL LIMIT 500 硬编码导致数据被强行砍断 (MEDIUM | 后端)
- **缺陷位置：** `supply_management_service.py` L191
- **隐患：** 
  在 `list_delivery_records` 中硬编码了 `LIMIT 500` 且未提供任何分页机制。随着项目进行，发货记录一旦超过 500 条，较早期的记录会在数据库层被直接截断，在库管员过滤台账时，因先被 `LIMIT` 截断，再被 Python 过滤，将导致部分未确认记录无法被查询出来。

### 8. `get_next_shipment_sequence` 吞掉全部异常并返回 1 (MEDIUM | 后端)
- **缺陷位置：** `supply_management_service.py` L470-471
- **隐患：** 
  该函数在生成新车次时，包裹了 `except Exception: return 1`。这会在数据库瞬断、锁等待或格式不兼容时静默返回序号 1。供应商提交时便会生成重复的车次号，破坏后续车次分组和车牌号同步逻辑。

### 9. 计划保存与使用量保存缺失后端日期范围验证 (HIGH | 后端)
- **缺陷位置：** `workspace.py` L1030
- **隐患：** 
  API 接口在接收前端提交的计划或使用量列表时，完全没有校验传入的 `plan_date` 或 `usage_date` 是否落在可修改的日期窗口内。这导致外部恶意请求可以通过直接修改 API Body，篡改数周前或数月后的核心用料数据。

### 10. `plan_editable_days` 配置项前后端分离失效 (MEDIUM | 后端)
- **缺陷位置：** `demand_management_service.py` L25
- **隐患：** 
  后端 `build_plan_dates` 依然使用硬编码 `range(3)`，忽略了管理员在配置文件中动态设置的 `plan_editable_days` 值，导致前后端对计划填报的视图大小理解出现分歧。
