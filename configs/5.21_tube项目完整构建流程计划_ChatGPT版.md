## 1.1 给外部协作伙伴的 Phoenix 平台背景说明

为便于将本计划转发给其他协作伙伴（例如网页版 ChatGPT）共同讨论，先说明 Phoenix 当前并不是一个“单页面小工具”或“单项目系统”，而是一个已经运行中的**平台级 Web 应用**。新的 tube 项目不是从零搭一个全新站点，而是作为该平台内的一个新业务子系统接入。

### Phoenix 当前的整体形态

Phoenix 目前是一个“统一登录 + 项目选择 + 多项目模块挂载”的平台式应用，核心特征如下：

1. 用户先登录平台；
2. 登录后进入项目选择页；
3. 根据账号权限进入不同项目空间；
4. 各项目在同一平台壳层下运行，但各自拥有独立页面、接口、数据配置与业务逻辑；
5. 平台统一提供认证、权限、路由壳层、公共组件、部署体系与部分通用服务。

也就是说，Phoenix 不是“tube 专用系统”，而是一个已经承载多个业务模块的主站。

### 前端基本情况

Phoenix 前端基于：

1. `Vue 3`
2. `Pinia`
3. `Vue Router`
4. `Vite`

前端根入口为：

```text
frontend/src/main.js
```

全局路由入口为：

```text
frontend/src/router/index.js
```

前端目前采用“平台页面 + 项目子目录”的组织方式：

```text
frontend/src/pages/
frontend/src/projects/
```

其中：

1. `pages/` 负责登录、项目选择、项目入口跳转等全局页面；
2. `projects/` 下每个目录对应一个具体项目模块；
3. 新项目如果是通用型页面，可复用既有动态页面；
4. 新项目如果是独立业务系统，则在 `frontend/src/projects/<project_key>/` 下建设专属页面，并在总路由中挂载。

### 后端基本情况

Phoenix 后端基于：

1. `FastAPI`
2. 统一 `/api/v1` 路由前缀
3. 项目级路由注册机制

后端根入口为：

```text
backend/main.py
```

总路由入口为：

```text
backend/api/v1/routes.py
```

项目路由注册表为：

```text
backend/api/v1/project_router_registry.py
```

后端目前采用“平台 API + 项目目录”的结构：

```text
backend/api/v1/
backend/projects/
backend/services/
backend/db/
backend/sql/
```

其中：

1. `api/v1/` 提供平台级通用接口与统一挂载逻辑；
2. `projects/` 下每个目录对应一个子项目后端实现；
3. 每个项目通过注册表挂载到统一路径：

```text
/api/v1/projects/<project_key>
```

### Phoenix 现有项目格局

根据当前仓库和项目清单，Phoenix 已经存在多个项目模块，典型包括：

1. `daily_report_25_26`
   - 当前平台的主干型项目；
   - 承担日报填报、审批、展示、分析、看板等核心能力；
   - 也是大量通用组件、样式、交互模式的物理样板来源。

2. `monthly_data_show`
   - 一个完成度较高的独立工作台型项目；
   - 包含“月报导入工作台”和“月报查询工具”；
   - 非常适合复用其工作台式布局、筛选区、结果区、导入导出区、交互节奏。

3. `monthly_data_pull`
   - 月报拉取工作台；
   - 更偏向数据处理和工作流工具型页面。

4. `daily_report_spring_festval_2026`
   - 春节专题看板型项目；
   - 适合作为“独立专题页面 + 独立路由 + 项目边界清晰”的样板参考。

5. `page_showcase`
   - 辅助型项目；
   - 用于展示或扫描页面资源，不是主业务核心。

### Phoenix 的项目清单驱动方式

Phoenix 存在一个很关键的项目总配置文件：

```text
backend_data/shared/项目列表.json
```

它至少承担以下职责：

1. 定义平台有哪些项目；
2. 定义各项目名称；
3. 定义项目有哪些页面入口；
4. 为前端项目选择页提供展示数据；
5. 为后端项目页面列表接口提供元数据。

这意味着：**一个新项目要接入 Phoenix，不只是写几张页面，还要先成为平台认可的“项目”。**

### tube 项目在 Phoenix 中的定位

新的 `insulation_pipe_supply_2026` 项目，定位不是“独立部署的新站点”，而是：

1. Phoenix 主站中的一个新项目；
2. 业务上独立，但技术上复用主站底座；
3. 拥有自己的前端目录、后端目录、配置和业务数据；
4. 通过平台统一登录、统一权限体系、统一项目入口进入。

因此，tube 项目与 Phoenix 的关系可以理解为：

```text
Phoenix = 平台主站 / 容器
tube = 平台中的一个新业务模块 / 子系统
```

### tube 项目将如何与 Phoenix 衔接

#### 1. 项目入口层衔接

tube 项目已经在项目清单中定义了项目键：

```text
insulation_pipe_supply_2026
```

并预设了 4 个入口：

1. `dashboard`
2. `supply_management`
3. `demand_management`
4. `warehouse_management`

这 4 个 key 当前应视为 **项目既定页面 key**，后续前端页面、后端接口、权限配置、文档说明和讨论口径，均以这组 key 为准推进，不再基于旧理解另行改名。

后续需要做的，是把这 4 个既定入口真正落成前端页面和后端接口，而不是只停留在配置层。

#### 2. 前端层衔接

我计划在以下目录建设 tube 项目前端：

```text
frontend/src/projects/insulation_pipe_supply_2026/
```

并在总路由中为 4 个入口补充正式路由，让用户可从 Phoenix 平台内进入：

1. 项目工作台；
2. 供给侧页面；
3. 需求侧页面；
4. 库管台账页面；
5. 全局看板页面。

前端会尽量复用 Phoenix 现有：

1. 头部和面包屑组件；
2. 通用主题和布局；
3. 权限获取方式；
4. API 请求封装；
5. 工作台型页面结构。

#### 3. 后端层衔接

我计划在以下目录建设 tube 项目后端：

```text
backend/projects/insulation_pipe_supply_2026/
```

并将项目路由正式注册进：

```text
backend/api/v1/project_router_registry.py
```

使其通过统一前缀暴露接口：

```text
/api/v1/projects/insulation_pipe_supply_2026/...
```

这样做的好处是：

1. 与 Phoenix 现有项目挂载方式完全一致；
2. 登录认证、权限依赖、项目路由边界都能复用；
3. 后续维护不会形成“平台内一个特殊异类项目”。

#### 4. 权限体系衔接

tube 项目第一阶段不准备单独建权限系统，而是复用 Phoenix 已有账户与权限体系。

也就是说：

1. 登录仍走平台已有认证；
2. 用户信息仍由平台已有账户文件或权限文件驱动；
3. tube 项目只是在平台权限模型上增加本项目的可见范围和动作范围；
4. 不另起一套完全割裂的用户系统。

这是 V5.0 方案里一个很重要的降复杂度决策。

#### 5. 数据层衔接

tube 项目不会把所有数据都塞回原有日报项目，而是采用“项目独立配置 + 项目独立业务表 + 平台共享鉴权”的接入方式。

即：

1. 静态/低频配置：`tube_config.json`
2. 核心业务数据：独立 `tube_*` 表
3. 鉴权与项目入口：继续走平台共享机制

这种方式既保持了 tube 项目的独立性，也避免了重新造一套独立主站。

### tube 项目对 Phoenix 的复用关系

从工程实现角度，tube 项目并不是“重新发明一套系统”，而是站在 Phoenix 现有成果上扩展。

我准备重点复用的既有能力包括：

1. 平台登录和会话体系；
2. 项目列表与项目入口机制；
3. 前端全局布局和视觉骨架；
4. API 请求封装和错误处理模式；
5. 后端项目级路由挂载机制；
6. 权限解析方式；
7. 文档与进度留痕方式；
8. 构建、部署与本地验证链路。

### 这对外部协作伙伴意味着什么

如果外部协作伙伴要共同讨论或完善 tube 项目方案，需要先建立一个正确前提：

1. 这里不是从零搭新站；
2. 也不是做一个孤立 demo；
3. 而是在一个已有平台中新增一个真实可运营的业务子系统；
4. 设计时必须考虑与 Phoenix 的目录结构、路由、权限、配置、文档规范和交付方式保持一致。

换句话说，tube 项目的最佳建设方式不是“另起炉灶”，而是：

```text
严格贴合 Phoenix 平台现有工程范式进行增量扩展
```

这也是本计划后续所有实施步骤的基础前提。

## 2. 总体实施原则

### 2.1 总原则

1. 先跑通主流程，再补强体验层。
2. 先固化数据和状态机，再做页面和视觉。
3. 先做一阶段闭环，不把二期能力混入首版。
4. 能复用 Phoenix 现有能力的地方不重复造轮子。
5. 页面入口按你当前已定义的 4 个入口推进，不额外改成“大一统单页系统”。

### 2.2 首版交付边界

本轮构建只覆盖 V5.0 文档第十四章“第一阶段建设范围”，不主动把以下内容混入首版：

1. 运输车辆、司机、路线管理；
2. 电子签名、图片签收、定位打卡；
3. 复杂审批流；
4. 正式确认文件自动生成；
5. 完整退库建议与调剂闭环；
6. ERP / OA / 财务系统深度集成；
7. 独立异常闭环系统；
8. 独立库存快照系统；
9. 智能调度算法。

## 阶段 7：权限接入与角色约束

### 目标

在 Phoenix 现有权限体系里，把 tube 项目真正限制到“谁能看什么、谁能动什么”。

## 5. 我认为最关键的几个技术决策点

## 5.1 `tube_delivery` 是否完全做成生命周期主表

我的判断：应该做，而且必须做。

原因：

1. 这是 V5.0 的核心收敛点；
2. 能显著降低第一阶段查询复杂度；
3. 非常适合当前平台快速落地；
4. 看板、台账、超时预警都更好做。

## 5.2 库存是否实时计算

我的判断：第一阶段必须实时计算，不单独建库存余额表。

原因：

1. 方案已经明确；
2. 库存来源清晰：`received_qty - usage_qty ± adjustment`；
3. 先避免双写库存表带来的对账复杂度。

## 5.3 前端先做哪个入口

我的判断：先做 `production_allocation`。

原因：

1. 计划和使用数据都由需求侧产生；
2. 到货确认也在需求侧；
3. 这是整个业务链的前半核心。

## 5.4 是否先做花哨看板

我的判断：不应优先做大屏美化，应先做表格化强可用页面。

原因：

1. 这个项目首版的价值来自流程数字化和数据清楚；
2. 不是来自视觉炫技；
3. 先把台账、风险、状态做清楚，后面再美化最稳。

为了减少后续返工，我建议你优先确认以下问题：

5. 发货记录是否允许撤销或回退；

7. 历史计划和历史使用填报是否允许修改；
8. 看板默认是“风险优先”还是“汇总优先”；
9. `dashboard` 是否只查看不操作；
10. `warehouse_management` 页面对最终用户是否要在标题、副标题和说明文案中反复强调“库管员管理入口 / 库管台账”定位。
