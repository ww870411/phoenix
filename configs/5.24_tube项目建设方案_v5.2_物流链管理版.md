# 5.21_tube项目建设方案_v5.2_物流链管理版

**模块名称：** 保温管物流链数字化管理模块  
**内部代号：** tube项目  
**依托平台：** 凤凰平台 / 智算平台现有技术底座  
**版本：** V5.2  
**本版定位：** 在 V5.1 建设方案基础上，按当前真实实现、真实配置、真实联调结果修正后的执行型业务方案  
**适用对象：** 公司领导、规划发展部、经济运行部、供给主体 / 管厂、库管人员、现场负责人、施工单位、系统开发人员

---

## 一、本版修订说明

本方案是在 V5.1 版《保温管物流链数字化管理模块建设方案》基础上，根据当前代码实现、配置实际状态和近期联调修正结果形成的 V5.2 版。

V5.2 的重点不是继续扩展需求，而是把已经被实现改变的口径收回到真实状态，避免“方案仍写旧规则、系统却按新规则运行”。

本版核心修订点如下：

1. 明确当前运行配置主源为 `tube_config.json`。
2. 明确 `show_date`、`plan_start_date`、`plan_editable_days` 当前都由配置文件维护，其中 `plan_start_date` 控制采集窗口，`show_date` 控制展示口径。
3. 明确施工单位映射不再使用独立 `construction_assignments`，而是直接内嵌在 `construction_units.station_ids`。
4. 明确基准量主数据当前已收口到 `tube_config.json -> baseline_presets`。
5. 明确 `tube.tube_baseline_quantity` 不再作为运行时依赖表。
6. 明确“滚动三日计划量”当前以 `plan_start_date` 为采集窗口起点，而展示层看到的滚动三日计划量以 `show_date` 为展示窗口起点。
7. 明确当前现场库存在已落地实现中按 `总到货 - 总使用量` 计算。
8. 明确“三日净缺口”当前只扣减 `已发货待到货量`，不再额外扣减 `已到货待接收量`。
9. 明确“在途时长”在业务上到“确认到货”时截止。
10. 明确 `cancelled` 状态不显示在途时长。
11. 明确当前项目已从“待建设方案”进入“已落地主流程骨架后的继续完善阶段”。

## 二、建设背景

老旧供热管网升级改造项目涉及换热站数量多、保温管型号多、供应主体不止一个，且不同型号的供应、发货、到货、施工接收、库管留痕和现场使用通常由不同角色在不同时间完成。

如果缺少统一数字化链路，会持续出现以下问题：

1. 三日计划、发货、到货、接收、库存、使用之间缺少统一口径；
2. 发货记录和现场状态之间难以追溯；
3. 短期需求与现场剩余之间缺少可计算结果；
4. 到货、接收、库管手续之间容易断链；
5. 方案文档、页面显示和后端算法一旦口径不一致，系统就会表面可用、实则误导。

因此，本模块的目标不是单纯做一个发货登记页，而是围绕“计划 -> 发货 -> 到货 -> 接收 -> 库管 -> 使用 -> 缺口”的业务链路，形成一套可记录、可计算、可查询、可追溯的轻量化系统。

## 三、项目定位

### 3.1 平台定位

本模块是 Phoenix 平台中的项目级子系统，不是独立站点。

当前依托平台统一提供：

1. 登录与会话管理；
2. 项目入口挂载；
3. 页面级权限过滤；
4. 项目级后端路由；
5. 项目数据目录与配置目录。

### 3.2 管理定位

本模块当前定位为：

1. 物流链状态记录系统；
2. 短期需求与供给节奏辅助系统；
3. 多角色确认留痕系统；
4. 现场库存与缺口计算系统；
5. 看板与风险提示的前置数据底座。

### 3.3 核心原则

1. 第一阶段优先跑通主流程；
2. 口径以真实实现为准；
3. 配置统一、字段统一、算法统一；
4. 页面展示优先复用后端统一结果；
5. 不为满足形式完整而保留已失效设计。

## 四、建设目标

### 4.1 管理目标

1. 建立从三日计划到现场使用的完整物流链台账；
2. 建立各换热站、各型号的短期需求、发货、到货、库存、使用和缺口的统一查看口径；
3. 支撑供给主体查看需求并安排发货；
4. 支撑现场负责人、施工单位、库管人员按角色完成各自确认动作；
5. 为领导层提供汇总看板与风险提示基础数据。

### 4.2 系统目标

1. 配置可维护；
2. 计划可录入；
3. 发货可登记；
4. 到货、接收、库管可确认；
5. 库存与缺口可自动计算；
6. 关键时间、状态和异常可追溯；
7. dashboard 可在下一阶段正式承接汇总展示。

## 六、业务建模假设与运行前提

### 6.1 时间周期假设

1. 统计周期仍按自然日 `0:00 - 24:00`；
2. 每日 `8:00` 前填报上一自然日实际使用量。

### 6.2 滚动三日计划窗口假设

V5.2修正：

1. “滚动三日计划量”不是单纯由系统当天日期自动推算；
2. 采集层以 `plan_start_date` 为三日窗口起点；
3. 展示层以 `show_date` 为三日窗口起点；
4. 因此方案、页面与配置必须承认“采集窗口”和“展示窗口”是两套不同日期控制，不能再混用。

### 6.3 施工单位映射假设

1. 施工单位可绑定多个换热站；
2. 其绑定范围不要求与现场负责人完全一致；
3. 权限校验以配置中的站点范围为准。

### 6.4 配置运行假设

1. `tube_config.json` 当前是实际运行主配置；
2. 全局管理页负责结构化维护；
3. 手改配置文件后，页面应重新拉取最新值。
4. 施工单位与换热站关系当前直接维护在 `construction_units.station_ids` 中。
5. `plan_start_date` 用于控制计划录入窗口和实际使用量采集日期；
6. `show_date` 用于控制 dashboard、汇总页和各业务页对外展示口径。

## 七、总体业务流程

### 7.1 全流程链条

```text
基准量 / 计划采购量
        ↓
滚动三日计划
        ↓
发货登记
        ↓
已发货待到货
        ↓
到货确认
        ↓
已到货待接收
        ↓
施工接收
        ↓
已接收待库管确认
        ↓
库管确认
        ↓
已完成
        ↓
每日实际使用
        ↓
库存 / 缺口 / 风险提示
```

### 7.2 计划管理流程

1. 现场负责人在其换热站范围内填报三日计划；
2. 系统按 `plan_start_date` 生成滚动三日采集窗口；
3. 计划以长表连续记录，不是“只保存最近三天”。

### 7.3 发货与多方确认流程

1. 供给主体登记发货；
2. 发货后进入 `pending_arrival`；
3. 现场负责人确认到货后进入 `pending_receive`；
4. 施工单位确认接收后进入 `pending_warehouse`；
5. 库管确认后进入 `completed`；
6. 仅 `pending_arrival` 允许撤销，撤销后为 `cancelled`。

### 7.4 每日实际使用流程

1. 现场负责人填报上一自然日实际使用量；
2. 实际使用量默认采集 `plan_start_date - 1` 对应自然日的数据；
3. 展示层实际使用、库存、累计量等统计以 `show_date` 作为推进确认标识。

## 八、关键业务规则

### 8.1 全部需求共享展示规则

供给主体可以查看全部需求，但只允许操作本主体发货记录。

### 8.2 设计用量与计划采购量规则

V5.2修正：

1. 业务上这两类数据仍属于基础控制数据；
2. 但当前运行实现已不再依赖数据库基准量表；
3. 第一阶段真实生效口径已收口到 `baseline_presets`。

### 8.3 滚动三日计划规则

1. 三日计划按三天分日记录；
2. 采集层三日窗口当前以 `plan_start_date` 为起点；
3. 展示层滚动三日计划量当前以 `show_date` 为起点；
4. 供给侧缺口计算、需求侧查看和 dashboard 汇总均应复用展示层的统一滚动三日计划量。

### 8.4 发货规则

1. 发货记录必须包含主体、换热站、型号、数量、发货时间；
2. 发货撤销只允许在 `pending_arrival`；
3. 撤销保留原记录、撤销人、撤销时间、撤销原因。

### 8.5 到货、接收与库存入账规则

V5.2修正：

1. 业务理想口径曾认为“库存以施工接收入账”；
2. 但当前实际实现中，库存计算已按 `总到货 - 总使用量` 运行；
3. 因此方案文档在本版本中以“当前已落地实现口径”为准；
4. 后续若要改回“按施工接收入账”，应作为明确的下一轮业务改动，而不是继续在文档中保留旧定义。

### 8.6 每日实际使用规则

1. 每日实际使用由现场负责人填报；
2. 第一阶段不开放历史修改。

### 8.7 权限与操作范围规则

1. 现场负责人：计划、到货确认、实际使用；
2. 施工单位：施工接收；
3. 供给主体：发货、撤销；
4. 库管：库管确认；
5. Global_admin：全局可见、全局可维护。

### 8.8 库管台账规则

库管页只负责手续闭环确认，不承接前序动作替代入口。

### 8.9 备注与异常规则

第一阶段仍通过备注、异常标记和操作留痕处理，不单独建设复杂异常闭环。

## 九、关键计算口径

### 9.1 基础数量

```text
设计用量 = 规划发展部按设计文件或技术口径确认的保温管需求量

计划采购量 = 规划发展部按现场管理和采购组织口径预计需要采购 / 供应的数量

累计发货量 = 供给主体已发货数量合计

已发货待到货量 = 已发货但尚未由现场负责人确认到货的数量

已到货待接收量 = 已由现场负责人确认到货但尚未由施工单位确认接收的数量

已接收待库管确认量 = 已由施工单位确认接收、尚未完成库管确认的数量

累计到货确认量 = 现场负责人确认到货数量合计

累计施工接收量 = 施工单位确认接收数量合计

累计实际使用量 = 每日实际使用量合计
```

V5.2补充说明：

1. 上述基础数量仍应作为完整方案口径保留，不因第一阶段页面是否全部展示而删除；
2. 当前系统已真实展示或参与汇总计算的主要量包括：
   - 三日计划量
   - 已发货待到货量
   - 已到货待接收量
   - 已接收待库管确认量
   - 累计到货相关汇总
   - 累计实际使用量
3. 当前库存量的真实实现口径为：

```text
当前现场库存量 = 总到货量 - 累计实际使用量
```

4. 这里的“总到货量”当前按已进入到货后的物流记录汇总计算；
5. 因此，V5.2 在保留“累计施工接收量”这一完整业务口径的同时，明确当前运行实现使用的是“总到货 - 总使用量”。

### 9.2 剩余量与完成率

```text
设计剩余量 = 设计用量 - 累计实际使用量

计划采购剩余量 = 计划采购量 - 累计施工接收量

设计完成率 = 累计实际使用量 / 设计用量

采购接收执行率 = 累计施工接收量 / 计划采购量

领用消化率 = 累计实际使用量 / 累计施工接收量
```

V5.2补充说明：

1. 这组口径仍属于方案层必须保留的完工复盘框架；
2. 第一阶段当前页面未全部正式展示这些指标，但后续 dashboard 和完工复盘仍应以这些定义为基础；
3. 若某换热站存在“已到货但未接收”的数量较大，则 `采购接收执行率` 与“已发货进度”应区分理解，不能混同。

### 9.3 滚动三日缺口

```text
滚动三日计划量 = 第1天计划 + 第2天计划 + 第3天计划

现场硬缺口 = max(滚动三日计划量 - 当前现场库存量, 0)

三日净缺口 = max(滚动三日计划量 - 当前现场库存量 - 已发货待到货量, 0)
```

V5.2确认：

1. 采集层三日窗口以 `plan_start_date` 为起点；
2. 展示层 `滚动三日计划量` 以 `show_date` 为起点；
3. `现场硬缺口` 仍保留为方案层概念，用于判断当前库存是否足以覆盖三日需求；
4. `三日净缺口` 当前只扣减 `已发货待到货量`；
5. `已到货待接收量` 不再作为净缺口独立扣减项；
6. 原因不是该数量不重要，而是当前库存口径已基于“总到货量”统计，若再单独扣减 `已到货待接收量`，会产生重复扣减；
7. `已到货待接收量`、`已接收待库管确认量` 仍应在页面或看板中单独展示，以保留物流分段透明度。

### 9.4 确认时效指标

```text
到货确认耗时 = 现场负责人确认到货时间 - 发货时间

施工确认耗时 = 施工单位确认时间 - 现场负责人确认到货时间

库管确认耗时 = 库管确认时间 - 现场负责人确认到货时间
```

1. 发货后超过 12 小时未确认到货，可提示；
2. 到货确认后超过 6 小时未施工接收，可提示；
3. 施工接收后库管未确认，可提示。

V5.2补充说明：

1. 这组指标仍是时效管理和超时预警的基础口径；
2. 当前代码层已具备发货时间、到货确认时间、施工接收时间、库管确认时间等关键时间字段；
3. dashboard 与风险提示后续应围绕这一组指标展开。

### 9.5 运输时长与完工复盘指标

```text
在途时长（未确认到货） = 当前时间 - 发货时间

在途时长（已确认到货） = 确认到货时间 - 发货时间

设计偏差量 = 累计实际使用量 - 设计用量

计划采购偏差量 = 累计实际使用量 - 计划采购量

剩余库存量 = 当前现场库存量

剩余待处理量 = 已完工状态下仍未使用或未退库的现场库存量
```

V5.2确认：

1. “在途时长”只统计运输过程；
2. 一旦现场负责人确认到货，在途计时立即截止；
3. `cancelled` 状态不显示在途时长；
4. 完工退库建议表第一阶段仍不建设，但“设计偏差量、计划采购偏差量、剩余库存量、剩余待处理量”等复盘指标不应从方案中删除，后续 dashboard 和二期分析仍需使用。

## 十、数据关系设计

### 10.1 数据关系总体思路

当前仍采用“配置文件 + 核心业务表 + 统一状态机”的轻量结构。

### 10.2 配置文件设计

建议设置模块配置文件：

```text
tube_config.json
```

主要保存以下内容：

| 配置项 | 内容 | 说明 |
|---|---|---|
| show_date | 当前展示日期 | 用于控制 dashboard、汇总页与业务页对外确认展示到哪一天。 |
| plan_start_date | 计划起始日期 | 用于控制滚动三日计划采集窗口，并决定实际使用量默认采集 `plan_start_date - 1`。 |
| plan_editable_days | 可编辑计划天数 | 当前用于约束计划录入边界。 |
| supply_entities | 供给主体 / 管厂 | 记录主体名称、联系人、账号关联、展示状态等。 |
| demand_entities | 换热站 / 需求主体 | 记录换热站名称、区域、施工状态等。 |
| pipe_models | 保温管型号字典 | 记录型号、单位、排序等信息。 |
| manager_assignments | 现场负责人绑定关系 | 记录现场负责人与换热站范围。 |
| construction_units | 施工单位配置 | 当前同时记录施工单位基础信息和 `station_ids` 映射范围。 |

| 超计划采购接收 | 累计接收 / 到货明显超过计划采购量 | 可能存在计划偏低或供应超量。 |
| 完工余料 | 已完工点位仍有库存 | 后续需要退库、调剂或说明。 |
| 数量差异 | 到货确认数量或施工接收数量小于上一环节数量 | 可能存在少发、破损、争议或拒收。 |

## 十九、结论

V5.2 版方案的重点不是提出更多新功能，而是把 tube 项目从“讨论中的建设方案”推进为“既保留完整业务蓝图、又与当前真实系统一致的执行性方案”。

本版明确两层判断：

1. 从方案层看：
   - 本模块仍然是围绕“计划、发货、到货、接收、库管、库存、使用、缺口、预警、复盘”的完整物流链数字化管理系统；
   - 不应因为第一阶段部分展示尚未全部完成，就把这些完整业务口径从方案文档中删除。
2. 从实现层看：
   - 主配置已落地；
   - 主流程页面已落地；
   - 发货到库管闭环已打通；
   - 审计暴露的核心问题已完成第一轮修复；
   - 当前主要工作已从“能否落地”转为“如何统一口径、补完 dashboard、完成联调回归”。

因此，后续工作应以本版为准：

1. 在方案层保持完整业务边界；
2. 在实现层持续校准文档与代码；
3. 优先继续完善 dashboard、回归边界场景、清理历史残留口径；
4. 再进入运输分析、退库建议、异常闭环和考核等二期扩展能力。
