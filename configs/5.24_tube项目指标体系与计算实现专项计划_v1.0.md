### 0.1 项目是什么

`insulation_pipe_supply_2026` 是 Phoenix 平台中的一个项目级模块，主题是：

**2026年度保温管供需管理 / 物流链管理**

它不是独立站点，而是挂载在 Phoenix 平台中的一个业务子系统。  
当前系统围绕以下业务链条运作：

```text
基准量 / 计划采购量
    ↓
未来三日计划
    ↓
供给主体发货
    ↓
现场负责人确认到货
    ↓
施工单位确认接收
    ↓
库管确认留痕
    ↓
每日实际使用
    ↓
库存 / 缺口 / 时效 / 风险展示
```

## 1. 文档目的

本专项计划用于回答三个问题：

1. tube 项目到底有哪些关键计算指标需要建设；
2. 这些指标应该由前端、后端还是前后端协同完成；
3. 如何把关键计算口径从“散落在代码中的硬编码”逐步升级为“可见、可核对、可维护、可追溯”的体系。

## 2. 总体判断

### 2.1 核心结论

不是所有指标都应由后端统一计算，也不是前端能算的就都交给前端。

应按指标性质分为三类：

1. `A类：后端标准指标`
2. `B类：后端提供基准 + 前端动态展示指标`
3. `C类：纯前端展示指标`

### 2.2 设计原则

1. 业务真值、跨页面一致口径、将来会进入 dashboard / 导出 / 审计解释的指标，优先归后端；
2. 依赖浏览器当前时间、只服务展示体验、不作为业务真值的指标，优先归前端；
3. 需要“后端确定规则、前端实时滚动展示”的指标，走协同模式；
4. 后续配置化优先覆盖 `A类` 指标，不优先碰 `B类/C类`。

## 3. 指标分层模型

### 3.1 第一层：基础事实层

这层不是“公式指标”，而是计算所依赖的原始事实量。  
特点：

1. 来自数据库、配置或状态流转结果；
2. 不应由任意公式动态生成；
3. 是所有高层指标的输入。

当前核心基础事实包括：

1. `design_qty`
2. `purchase_plan_qty`
3. `plan_qty`
4. `shipped_qty`
5. `arrived_qty`
6. `received_qty`
7. `usage_qty`
8. `pending_arrival_qty`
9. `pending_receive_qty`
10. `pending_warehouse_qty`
11. `completed_qty`
12. `total_shipped_qty`
13. `total_arrived_qty`
14. `total_usage_qty`
15. `shipped_at`
16. `arrived_confirm_at`
17. `received_confirm_at`
18. `warehouse_confirm_at`
19. `status`
20. `plan_start_date`
21. `biz_date`

### 3.2 第二层：标准业务指标层

这层是“九、关键计算口径”的核心。  
特点：

1. 要跨页面统一；
2. 可能进入 dashboard；
3. 将来应优先进入配置化定义。

### 3.3 第三层：展示与交互层

这层是页面和看板展示所需的动态值。  
特点：

1. 依赖当前时间或本地时区；
2. 不作为业务真值落库；
3. 允许前端实时更新。

## 5. 实现策略

### 5.1 当前阶段不建议做的事

不建议直接把任意 Python/JS 公式写进配置文件然后动态执行。  
原因：

1. 安全边界差；
2. 调试困难；
3. 前后端解释容易漂；
4. 容易把“去黑箱”做成新的更复杂黑箱。

### 5.2 建议做的事

建议引入“受控指标定义层”，而不是“任意公式执行层”。

可选落地方式：

1. 在现有 `tube_config.json` 中新增 `metric_definitions`
2. 或拆出单独文件，例如：
   - `tube_metric_definitions.json`

建议优先使用**独立文件**，原因：

1. 避免把业务配置和指标规则混在一起；
2. 更适合后续单独版本管理；
3. 更适合将来给 dashboard、审计、业务方单独查看。

```json
      "category": "inventory",
      "kind": "backend_standard",
      "formula": "total_arrived_qty - total_usage_qty",
      "inputs": ["total_arrived_qty", "total_usage_qty"],
      "rounding": 2,
      "clamp_min": null,
      "enabled": true,
      "notes": "当前执行口径"
    },
    "net_gap_qty": {
      "label": "三日净缺口",
      "category": "gap",
      "kind": "backend_standard",
      "formula": "max(future_plan_qty - station_inventory_qty - pending_arrival_qty, 0)",
      "inputs": ["future_plan_qty", "station_inventory_qty", "pending_arrival_qty"],
      "rounding": 2,
      "clamp_min": 0,
      "enabled": true,
      "notes": "当前执行口径"
    }
  }
}
```

### 6.3 字段说明

1. `label`：中文名称
2. `category`：分类，如 `inventory / gap / timing / review`
3. `kind`：指标实现分层
4. `formula`：受控公式字符串
5. `inputs`：依赖的基础事实字段
6. `rounding`：保留精度
7. `clamp_min`：最小值约束
8. `enabled`：是否启用
9. `notes`：当前口径说明

---

## 7. 计算执行设计

### 7.1 后端执行原则

后端不直接 `eval` 公式，而是实现一个受控解析器，只允许：

1. `+ - * /`
2. `max / min`
3. 白名单字段名
4. 少量受控辅助函数

### 7.2 计算流程

建议流程：

1. 先取基础事实层数据
2. 再按指标依赖顺序计算标准业务指标
3. 再统一输出 API 响应
4. 前端只消费结果，不重新发明业务公式

### 7.3 输出规则

后端建议统一输出：

1. 原始基础字段
2. 标准业务指标字段
3. 指标元信息，可选：
   - label
   - unit
   - notes

## 12. 风险与控制

### 12.1 主要风险

1. 公式配置化后调试变复杂
2. 指标依赖顺序出错
3. 前后端过渡期口径双轨
4. 业务方误以为“所有规则都能无代码配置”

### 12.2 控制措施

1. 只对白名单指标配置化
2. 先做 A类，不碰权限和状态机
3. 指标定义文件必须写 `inputs` 和 `notes`
4. 每个指标保留测试样例
5. API 响应中保留必要中间量，便于审计

3. 基础事实仍由后端取数
4. 标准业务指标由后端统一计算

6. 最后让 dashboard、页面、文档都引用同一套指标口径
