  ### 二、 已完成进度中存在的不足与问题（深剖）
  #### 🚨 缺陷一：未来三日计划矩阵回显恒为  0 （API 键值错位致命 Bug）
  • 定位文件：
      • 后端 API：workspace.py
      • 后端服务：demand_management_service.py
  • 技术原理分析：
  在  workspace.py  获取未来三日计划矩阵的接口中，API 试图从  matrix 
  获取具体型号和日期的计划数据：
    record = matrix.get(pipe_model_id, {}).get(key)
  该逻辑假设  matrix  是一个嵌套结构： { pipe_model_id: { plan_date: record }
  } 。但实际上，底层的服务函数  list_plan_records  返回的是一个扁平化的一阶  
  Dictionary，其键值是用双冒号拼接的字符串：
    result[f"{_normalize_pipe_model_id(row['pipe_model_id'])}::{date_key}"] =
  { ... }
  由于这个结构错位，API 层调用  matrix.get(pipe_model_id)  时将永远返回  {}  
  （空字典），进而  .get(key)  永远返回  None 。
  • 产生后果：
即便用户在数据库中成功保存了计划数据，重新进入页面时，回显出来的计划量将永远是
  0 ，导致现场填报人误以为计划没有保存成功而进行重复提交或无法工作。

  #### 🚨 缺陷二：施工单位登录后下拉站列表全空，且确认接收触发  403 Forbidden
  阻断 Bug
  • 定位文件：
      • 后端服务：config_service.py
      • 后端 API：workspace.py
  • 技术原理分析：
  在系统鉴权逻辑中，所有需求侧接口都依赖  resolve_accessible_station_ids 
  函数来判断当前登录账号可以管理哪些换热站。
  然而，该函数只实现了针对  manager_assignments                              
  （现场负责人绑定关系）的站点过滤：
    manager_assignments = get_config_list(payload, "manager_assignments")
    # 循环解析并添加现场负责人名下的 station_ids ...
  代码中完全没有读取和解析  construction_units                               
  （施工单位配置）中的换热站绑定关系！这导致，当施工单位用户（如             
  construction_a ，角色为  tube_construction_unit ）登录并请求  demand-      
  management/options  时，后端返回的可用换热站列表（ stations ）必然为空  {} 
  。更严重的是，当施工单位在待确认物流中点击“施工接收确认”时，API
  会进行强校验：
    accessible_station_ids = resolve_accessible_station_ids(...)
    _ensure_station_access(delivery["station_id"], accessible_station_ids)
  由于可用换热站为空，此处必然抛出  403 Forbidden                            
  （当前账号无换热站的访问权限），直接将确认操作阻断！
  • 产生后果：

施工单位无法在下拉框中选择站点，也完全无法执行“施工接收”确认，整条发货、到货、
施工接收、库管手续的物流状态机彻底卡在第二步（已到货待接收）。

  #### 🚨                                                                    
  缺陷三：三日净缺口数学公式存在“双倍重复扣减”逻辑漏洞（重大业务风险隐患）
  • 定位文件：
      • 后端 API：workspace.py
      • 后端服务：supply_management_service.py
  • 数学与逻辑漏洞推导：
  系统在供给侧界面计算  net_gap_qty （三日净缺口）的公式为：
  $$\text{三日净缺口} = \max(\text{未来三日计划} - \text{在途管道量} -
  \text{现场库存量}, 0)$$其中，各变量在后端代码的累加范围如下：
      1. $\text{在途管道量 (inbound_pipeline_qty)} = \text{已发货待到货} +
      \text{已到货待接收 (pending_receive_qty)} + \text{已接收待库管
      (pending_warehouse_qty)}$
      2. $\text{现场库存量 (station_inventory_qty)} = \text{总到货量
      (total_arrived_qty)} - \text{累计使用量}$
      而在  list_arrival_aggregates  服务中，计算  total_arrived_qty  时的
      SQL 筛选条件为：
        WHERE status IN ('pending_receive', 'pending_warehouse', 'completed')
  致命的双倍扣减推导：
  当某批物资的状态为  pending_receive （已到货待接收） 或  pending_warehouse 
  （已接收待库管） 时：
      • 它被作为在途量累加到了  inbound_pipeline_qty  中；
      • 同时，它又因为符合  status IN ('pending_receive', 'pending_warehouse',
      'completed') ，被累加到了  total_arrived_qty （也就是                  
      station_inventory_qty ）中！
  在最终的减法算式中，这笔已发货/已到货的数量被同时扣减了两次！
      • 实例印证：
      假设未来三日计划为 $100$ 米，现场目前没有库存和使用。此时发货一笔 $40$
      米，并已由现场负责人确认到货（状态流转至  pending_receive ）。
          • 代码计算：
              • $\text{在途管道量} = 40$
              • $\text{现场库存} = \text{总到货}(40) - 0 = 40$
              • $\text{计算出的净缺口} = 100 - 40 (\text{在途}) - 40
              (\text{库存}) = 20$ 米。
          • 物理真实：
              • 实际在途加到货一共只有 $40$ 米，物理上的真实缺口应该是 $100 -
              40 = 60$ 米！
  • 产生后果：
计算出的“三日净缺口”将严重虚低。供给侧管厂看到缺口极小，会减少排产与发货，导致
现场实际施工面临严重的断货和停工风险！
