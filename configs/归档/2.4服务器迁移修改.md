# Phoenix 项目 - 服务器迁移与 NPM 适配修改小结 (2026-02-04)

为了将应用从旧服务器迁移到性能更强的新服务器，并接入 **Nginx Proxy Manager (NPM)** 进行多站点统一管理，我们对构建和部署配置文件进行了针对性修改。

## 核心逻辑转变
*   **过去**：Phoenix 容器独立负责 SSL 证书申请、续期和 HTTPS 强制重定向。
*   **现在**：NPM 负责所有域名的 SSL 终止、证书续期及 HTTPS 安全策略，Phoenix 容器在内网环境下仅处理纯 HTTP 请求。

---

## 1. 构建文件对比
**文件：** `docker-compose.server.yml` (原) vs `docker-compose.server_new_server.yml` (新)
**阶段：** 本地构建 (Local Build)

| 修改项 | 修改内容 | 修改原因 (Why) |
| :--- | :--- | :--- |
| **构建参数 (Args)** | **新增** `HTTP_ONLY: "true"` | **核心修改。** 确保 Web 镜像使用 `nginx.http-only.conf`。禁用容器内部 Nginx 的 HTTPS 强制跳转，避免与 NPM 形成重定向死循环。 |
| **端口映射** | `80:80`, `443:443` 改为 `8001:80` | **避免端口冲突。** 在本地或测试环境下，防止占用宿主机的标准 80/443 端口（这些端口通常预留给 NPM）。 |
| **服务裁剪** | **删除** `web-http` 和 `certbot` 服务 | **简化架构。** 证书管理已移交给外部 NPM，构建过程中不再需要独立的 Certbot 验证环境。 |
| **卷挂载清理** | **删除** `certbot_etc` 和 `certbot_www` | **清理冗余。** 既然容器内部不再运行 Certbot，相关用于存储验证信息的磁盘卷也随之移除。 |

---

## 2. 部署文件对比
**文件：** `lo1.yml` (原) vs `lo1_new_server.yml` (新)
**阶段：** 服务器部署 (Server Deployment)

| 修改项 | 修改内容 | 修改原因 (Why) |
| :--- | :--- | :--- |
| **端口映射** | `80:80`, `443:443` 改为 `8001:80` | **适配 NPM 转发。** NPM 将外部 HTTPS 请求解密后，通过宿主机的 `8001` 端口转发给 Web 容器。不再需要容器监听 443。 |
| **证书挂载移除** | **删除** 所有证书路径挂载 | **职责分离。** 容器不再需要读取 SSL 证书文件，安全层已完全由宿主机的 NPM 接管。 |
| **网络流量模式** | 容器内仅运行 HTTP 服务 | **性能与兼容性。** 减轻容器负载，且符合“总管（NPM）+ 员工（各应用容器）”的分布式部署标准。 |

## 3. 流量处理流程示意
修改后，请求的完整链路如下：

1. **用户** 发起 `HTTPS://platform2.smartview.top` 请求。
2. **NPM (宿主机)** 接收请求，利用自带的 SSL 证书进行解密（SSL Termination）。
3. **NPM** 将流量以 **HTTP** 协议转发至 `http://localhost:8001`。
4. **Phoenix-Web (容器)** 接收 HTTP 请求并展示前端页面。
5. **API 请求** 通过容器内部网络转发至 **Phoenix-Backend (容器)** 的 8000 端口。

## 4. 常见疑问解答 (FAQ)

### Q1: 改动之后，新的镜像还存在 Nginx 服务器吗？
**存在。**
*   **角色转变**：它从“互联网前哨”变成了“内部宿管”。
*   **必要性**：前端 SPA 应用需要一个 Web 服务器（Nginx）来托管静态资源（HTML/JS/CSS），并根据 Nginx 配置处理 `/api` 的反向代理。没有它，容器无法独立提供页面服务。

### Q2: “8001:80”具体是什么意思？如果 NPM 已占用了 80/443，那这里的 80 是什么？
*   **Docker 映射逻辑**：格式是 `宿主机端口 : 容器端口`。
*   **容器端口 (80)**：这是容器内部 Nginx 启动时写死监听的端口。无论宿主机怎么映射，容器内部的世界里，它就在 80 端口等着。
*   **宿主机端口 (8001)**：这是你在新服务器上给这个容器留出的“专用通道”。
*   **链路**：用户访问 NPM (`443`) -> NPM 转发给宿主机 `8001` -> 流量自动进入容器 `80`。

### Q3: 为什么构建/部署两侧都需要设置存储卷和网络接口？
*   **关于存储卷**：
    *   **容器内路径 (如 /app/data)**：是硬编码在代码里的，不能变。
    *   **宿主机路径**：构建侧写是为了本地测试有地方存数据；部署侧写是为了在服务器上持久化生产数据。
*   **关于网络接口**：
    *   **虚拟路由器**：`networks` 让 `web` 和 `backend` 容器能像在一个局域网里一样，通过对方的名字相互通信。
    *   **双侧设置**：保证了这套“微服务架构”在任何环境下（本地开发、CI/CD 构建、服务器运行）都能闭环，逻辑不崩溃。

# 📅 2026-02-04 迁移复盘与技术总结

以下是对本次从旧服务器迁移至新服务器（ARM架构 + NPM管理）的全程回顾与技术沉淀。

## 一、 核心理念：服务器迁移的本质

服务器迁移不仅仅是“文件的搬运”，而是**运行环境的重构与适配**。在本次迁移中，我们遵循了以下三大原则：

1.  **职责分离 (Decoupling)**：将 SSL 证书管理、域名入口管理从应用容器中剥离，移交给专用的网关（Nginx Proxy Manager），应用回归纯粹的业务逻辑。
2.  **环境适配 (Adaptation)**：识别源环境（x86 Windows）与目标环境（ARM Linux）的差异，通过交叉编译解决架构不兼容问题。
3.  **配置隔离 (Isolation)**：通过创建独立的新配置文件（`_new_server` 后缀），确保在不破坏原有生产环境配置的前提下，安全地进行新环境的调试与部署。

## 二、 有效改动大纲

本次迁移涉及 4 个关键文件的修改与创建：

1.  **构建清单**：`docker-compose.server_new_server.yml` (新建)
2.  **Nginx 配置**：`deploy/nginx.http-only.conf` (修改)
3.  **部署清单**：`lo1_new_server.yml` (新建)
4.  **自动化脚本**：`lo1_new_server.ps1` (新建)

## 三、 技术详解：改了什么？为什么改？

### 1. 架构升级：接入 Nginx Proxy Manager (NPM)
*   **改动**：在部署文件中移除了 `certbot` 容器，删除了 SSL 证书挂载卷，停止监听 443 端口。
*   **原因**：旧架构是“单体应用自管门锁”，每个应用都要自己申请证书。新架构采用“大厦保安模式”，NPM 作为总管统一处理 HTTPS 加密解密，然后将干净的 HTTP 流量转发给应用。这使得一台服务器能轻松托管多个网站且互不冲突。

### 2. 解决重定向死循环：`HTTP_ONLY` 模式
*   **改动**：在 `docker-compose.server_new_server.yml` 中添加构建参数 `HTTP_ONLY: "true"`。
*   **原因**：NPM 已经负责了 HTTPS。如果容器内部的 Nginx 再强制跳转 HTTPS，就会出现 `NPM(HTTP) -> 容器(跳转HTTPS) -> 浏览器(重发) -> NPM` 的无限死循环。开启此模式让容器只处理 HTTP 请求，打破循环。

### 3. 解决 502 Bad Gateway：修正 `proxy_pass`
*   **改动**：将 `deploy/nginx.http-only.conf` 中的 `proxy_pass http://backend:8000$request_uri;` 修改为 `proxy_pass http://backend:8000;`。
*   **原因**：在 Nginx 反向代理配置中，不当使用 `$request_uri` 变量会导致路径拼接异常或变量解析失败，使得 Nginx 无法正确连接后端（Backend），从而报出 502 错误。标准写法更稳健。

### 4. 跨芯片架构适配：ARM64 交叉编译
*   **改动**：在 `docker-compose.server_new_server.yml` 中为服务添加 `platform: linux/arm64`。
*   **原因**：本地电脑是 x86 架构（Intel/AMD），新服务器是 ARM 架构（如 Ampere/Graviton）。如果不指定平台，构建出的 x86 镜像在 ARM 服务器上无法运行（报错 `exec format error` 或 `no matching manifest`）。此设置指示 Docker 在构建时启用 QEMU 模拟器生成 ARM 镜像。

### 5. 端口策略调整：`8001:80`
*   **改动**：将宿主机的 8001 端口映射到容器的 80 端口。
*   **原因**：宿主机的标准端口（80/443）已被 NPM 占用。我们需要开辟一个非标准端口（8001）作为“后门”，供 NPM 转发流量进入容器。

### 6. 自动化流维持：新脚本 `lo1_new_server.ps1`
*   **改动**：复制并修改原构建脚本，使其指向新的 `yml` 配置文件。
*   **原因**：为了不破坏旧服务器的维护流程，建立一套独立的脚本用于新环境的构建与推送，保证了新旧两套环境可以并存，降低了迁移风险。
