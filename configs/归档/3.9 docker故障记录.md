# 3.9 Docker 故障记录

日期：2026-03-09

## 一、问题背景

- 本应用长期部署在服务器上，通过 Docker 运行，之前可正常访问与登录。
- 服务器在 2026-03-08 至 2026-03-09 期间发生过一次意外重启。
- 重启后，用户发现服务器上的两个 Docker 环境发生错乱：
  - 原本长期使用的 `docker1` 被 `docker2` 取代；
  - 执行 Docker 命令时一度默认命中了 `docker2`；
  - 后续用户已停用 `docker2`，并将默认 Docker 切回 `docker1`。
- 切回后，部分 Phoenix 相关容器仍显示在运行，网站首页也可访问，但登录失效。

## 二、最初现象

- 用户访问网站时，登录页能够正常打开。
- 点击登录按钮后，长时间等待，最终失败。
- 用户提供的浏览器控制台记录见 `configs/3.9 登录错误.md`。

核心报错为：

```text
POST https://platform.smartview.top/api/v1/auth/login 504
```

返回内容为 Cloudflare 的 `504 Gateway time-out` 页面，而不是 Phoenix 自己的 JSON 错误。

## 三、第一阶段判断

基于前端和后端代码检查，先得到以下结论：

### 1. 前端请求链路正常

- 登录页文件：`frontend/src/pages/LoginView.vue`
- 登录 API 文件：`frontend/src/projects/daily_report_25_26/services/api.js`
- 前端登录实际请求固定为：

```text
POST ${API_BASE}/auth/login
```

- 浏览器报错已证明请求实际发到了：

```text
https://platform.smartview.top/api/v1/auth/login
```

因此：

- 不是前端按钮没触发；
- 不是登录页路由问题；
- 不是前端根本没发请求。

### 2. 后端登录代码逻辑本身不复杂

- 登录接口：`backend/api/v1/auth.py`
- 认证核心：`backend/services/auth_manager.py`

登录逻辑主要只做三件事：

1. 读取账号与权限配置；
2. 校验用户名密码；
3. 生成会话；

只有在勾选“记住登录”时，才会把会话写入 PostgreSQL 的 `auth_sessions` 表。

因此最初判断是：

- 如果返回 504，更像是部署链路问题；
- 不是典型的业务 401/404/500。

## 四、生产部署结构核对

仓库内与服务器相关的部署文件包括：

- `docker-compose.server.yml`
- `docker-compose.server_new_server.yml`
- `lo1_new_server.yml`

后续用户确认，服务器实际启动使用的是：

- `lo1_new_server.yml`

该文件的结构是：

1. `phoenix-web`
   - 暴露端口：`8001:80`
   - 仅作为 Nginx 静态站点与 `/api` 反代入口
2. `phoenix-backend`
   - 仅加入内部 Docker 网络，不直接对外暴露端口
3. `phoenix-db`
   - PostgreSQL
4. `phoenix_net`
   - Compose 内部网络逻辑名
   - 实际 Docker 中会自动命名为 `25-26_phoenix_net`

另外，服务器上还存在：

- `nginx_proxy_manager-app-1`

说明实际公网访问链路是：

```text
浏览器 -> Cloudflare -> Nginx Proxy Manager -> 宿主机 8001 -> phoenix-web -> phoenix-backend
```

## 五、服务器实测过程与结果

### 1. 容器与网络存在

用户最初提供的服务器输出显示：

- `phoenix-web`
- `phoenix-backend`
- `phoenix-db`

都在运行，且同属：

```text
25-26_phoenix_net
```

说明：

- 不是容器没启动；
- 不是 `web` 与 `backend` 根本不在一个网络。

### 2. `phoenix-web` 日志直接指向 upstream 连接超时

用户提供的 `phoenix-web` 日志中多次出现：

```text
upstream timed out (110: Operation timed out) while connecting to upstream
upstream: "http://172.19.0.3:8000/api/v1/auth/login"
```

这一步非常关键，说明：

1. `phoenix-web` 能解析出 `backend` 的容器 IP；
2. 但它在连接 `172.19.0.3:8000` 时超时；
3. 超时发生在“建立连接”阶段，而不是收到业务响应后。

### 3. `phoenix-backend` 最初日志为空

在最初的同一批测试中，用户记录：

- `phoenix-backend` 日志为空

由此一度推测：

- 可能后端容器虽然显示 `Up`，但实际服务没有监听 8000；
- 或者 `uvicorn` 父进程存活，但真正 worker 没起来。

### 4. 第二轮测试证明 backend 容器内部服务正常

后续用户继续执行了容器内自检。

结果如下：

- `docker top phoenix-backend`
  - 可见 `uvicorn backend.main:app --host 0.0.0.0 --port 8000 --workers 1 --reload`
- `docker logs phoenix-backend`
  - 出现：
    - `Uvicorn running on http://0.0.0.0:8000`
    - `Application startup complete`
- 容器内 Python 自检：
  - `127.0.0.1:8000/healthz` 正常
  - `172.19.0.3:8000/healthz` 也正常
  - `socket.connect_ex` 对 `127.0.0.1:8000` 与 `172.19.0.3:8000` 都返回 `0`

这一步把两个猜想排除了：

- 不是后端服务没启动；
- 不是后端只绑定到 loopback。

### 5. 第三轮测试证明“跨容器访问”异常

用户继续做了两个关键测试：

1. 从 `phoenix-web` 容器内访问：

```text
wget http://backend:8000/healthz
```

结果：超时

2. 新起一个临时 `busybox` 容器，加入同一网络，再访问：

```text
wget http://backend:8000/healthz
```

结果：仍超时

与此同时：

- `phoenix-backend` 容器内部访问自己是正常的；
- 其他容器访问 `phoenix-backend` 却超时。

因此可得出当时的核心结论：

```text
backend 应用本身是健康的，但 Docker bridge 网络内到 backend 容器的跨容器流量异常。
```

## 六、网络层排查结果

### 1. 旧网络存在脏状态

在停止 Phoenix 容器后，用户检查：

```text
docker network inspect 25-26_phoenix_net
```

结果显示：

- `Containers: {}`
- 但 `IPsInUse: 3`

这说明：

- 网络里已经没有挂着的容器；
- 但 IPAM 仍保留了占用记录；
- 网络状态不干净，符合“bridge 网络脏状态”的判断。

### 2. 旧网络已成功删除

随后用户执行：

```text
docker network rm 25-26_phoenix_net
```

删除成功。

之后 `docker network ls` 已确认该网络消失。

### 3. 重新启动时出现旧 network id 残留报错

用户再次执行：

```text
docker compose -f lo1_new_server.yml up -d
```

出现报错：

```text
failed to set up container networking: network 87e83a... not found
```

该错误表明：

- Docker 在为 Phoenix 容器建立网络时，仍引用了已经删除的旧 network id；
- 说明除了 bridge 网络脏状态外，还存在旧容器或 Docker 自身的网络元数据残留。

## 七、当前阶段结论

截至当前，故障可归纳为：

### 已确认正常的部分

1. 前端登录页可访问；
2. 前端登录请求确实发出；
3. `phoenix-backend` 应用本身可以启动；
4. 后端容器内部 `8000` 端口可访问；
5. `lo1_new_server.yml` 的整体部署结构逻辑正确；
6. 旧的 `25-26_phoenix_net` 已经成功删除。

### 已确认异常的部分

1. 删除前，`25-26_phoenix_net` 存在脏 IPAM 状态；
2. 跨容器访问 `phoenix-backend:8000` 超时；
3. 删除网络后，重新 `up -d` 时仍引用旧 network id；
4. 问题已经从应用层收敛到 Docker 网络/容器元数据层。

### 因此当前最合理的根因判断

```text
服务器重启 + Docker 环境切换后，Phoenix 所在的 Docker 网络与容器网络元数据发生错乱，导致：
1. bridge 网络脏状态；
2. 跨容器访问 backend 失败；
3. 后续重新建网时仍引用旧 network id。
```

## 八、当前不建议继续纠缠的方向

以下方向在本轮排查中已基本排除，不应继续优先投入：

1. 前端登录逻辑 bug
2. Phoenix 后端业务代码 bug
3. FastAPI 没启动
4. Uvicorn 没监听 8000
5. Nginx 配置语法错误

这些都不是当前主矛盾。

## 九、建议后续由 Docker/运维专家继续处理的重点

建议后续专家优先关注以下几项：

1. 删除并清理 Phoenix 残留容器元数据
   - `phoenix-web`
   - `phoenix-backend`
   - `phoenix-db`
2. 检查 Docker daemon 是否仍缓存旧 network id
3. 必要时重启 Docker 服务
4. 若仍异常，改为显式子网重建 Phoenix 网络，例如：

```yaml
networks:
  phoenix_net:
    driver: bridge
    ipam:
      config:
        - subnet: 172.30.250.0/24
```

5. 重新验证：
   - `phoenix-web -> backend:8000/healthz`
   - `curl http://127.0.0.1:8001/api/v1/auth/me`
   - 公网登录

## 十二、一句话总结

本次问题不是 Phoenix 代码本身坏了，而是服务器重启和 Docker 环境切换后，Phoenix 所在的 Docker 网络与容器网络元数据发生错乱，导致登录请求在 `phoenix-web -> phoenix-backend` 这一步无法正常建立连接，并进一步引发 504。

## 十三、深度根因分析：Snap Docker 与 Apt Docker 的冲突

经过进一步排查与技术对齐，确认了导致本次“网络幽灵”故障的底层原因：

### 1. 双重守护进程冲突 (Daemon Conflict)
服务器上同时存在两个 Docker 实例：
- **Apt Docker**: 用户长期使用且习惯的传统版本。
- **Snap Docker**: Ubuntu/Oracle Linux 系统中通过 `snap` 预装或误装的版本。

### 2. 事故发生链路
1. **意外重启**：服务器重启后，Linux 内核加载了两个 Docker 的相关内核模块。
2. **默认路径切换**：系统将 `docker` 命令默认绑定到了 `snap docker`。
3. **元数据污染**：
   - 虽然用户后续将路径切回了 `apt docker`，但 `snap docker` 在后台极可能已经启动或其留下的 **IPTables 防火墙规则**、**虚拟网桥 (docker0)** 已优先占据了宿主机内核。
   - 这导致两个 Docker 实例在抢夺对 Linux Bridge (网桥) 和路由表的控制权。
4. **IPAM 与 路由冲突**：
   - 在旧网段 (`172.19.0.x`) 下，数据包在发送时，虽然 `apt docker` 认为路径正确，但底层内核流量可能被 `snap docker` 残留的规则链拦截或误导，导致跨容器访问超时。

### 3. “换网段”方案为何有效？
修改 `lo1_new_server.yml` 中的子网（改为 `172.30.25.0/24`）是解决此类冲突的“降维打击”手段：
- **物理隔离**：新网段是 `snap docker` 完全未触碰过的领地。
- **强制刷新**：定义新子网会迫使 `apt docker` 向 Linux 内核申请创建全新的网桥和 IPTables 规则，绕过了所有旧的、脏的 IP 缓存和错误的路由路径。
- **路由重构**：这相当于在“被污染的地基”旁边盖了一座新房，避开了所有隐形的破坏者。

## 十四、最终修复与运维建议

1. **彻底清理 Snap Docker**：
   - 建议在服务器上执行 `sudo snap remove docker --purge`，确保系统中只有一个“大脑”。
2. **清理内核残留**：
   - 移除 `snap` 后重启 Docker 服务 (`sudo systemctl restart docker`) 以重置网桥状态。
3. **保持显式网段**：
   - 建议在生产环境中保留 `lo1_new_server.yml` 中的显式网段定义，这能显著提高容器网络的确定性，防止未来再次发生 IPAM 随机分配冲突。
