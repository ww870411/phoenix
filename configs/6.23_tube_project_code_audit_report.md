### 1. 库存计算公式严重错误

**问题描述**:  
后端库存计算使用 `received_qty`（施工接收量）而非 `arrived_qty`（到货确认量），导致库存统计不准确。

**影响范围**:
- 供给侧需求汇总接口 (`/supply-management/demand-summary`)
- Dashboard 数据看板
- 所有依赖库存数据的业务逻辑

**文档定义** (第10.1节):
```
现场可用在库 = 累计确认到货量 - 累计施工使用量 - 累计施工损耗量

到货后全面入账：从"发货"到"现场确认到货"之间的阶段定义为"在途"（对应 pending_arrival 状态）；
一旦执行了"确认到货"（arrived_confirm_at 不为空），所有后续流转状态其物资物理上已经运抵现场，
故均统一计入累计到货量。
```

**实际代码** (`supply_management_service.py:106`):
```python
SELECT
    station_id,
    pipe_model_id,
    SUM(
        CASE
            WHEN status <> 'cancelled'
                 AND COALESCE(arrived_qty, 0) > 0
                 AND arrived_confirm_at < :show_date
                THEN COALESCE(received_qty, arrived_qty, shipped_qty)  # ❌ 错误：优先使用 received_qty
            ELSE 0
        END
    ) AS total_arrived_qty
```

**问题分析**:
1. 当 `received_qty < arrived_qty` 时（少接收场景），库存会被**低估**
2. 处于 `pending_receive` 状态的物资（已到货但未施工接收）**不会被计入库存**
3. 这与文档描述的"到货后全面入账"**严重矛盾**

**修复方案**:
```python
# 修改 supply_management_service.py 第106行
THEN COALESCE(arrived_qty, shipped_qty)  # ✅ 使用到货确认量
```

**验证步骤**:
1. 创建测试发货单：发货 100 米
2. 确认到货：到货 95 米（少到货 5 米）
3. 施工接收：接收 90 米（少接收 5 米）
4. 查看库存：
   - 当前错误逻辑：库存 = 90 - 使用量
   - 正确逻辑：库存 = 95 - 使用量

**优先级**: 🔴 **P0 - 立即修复**  
**预计工作量**: 1小时  
**风险**: 高（影响核心业务数据准确性）

**修复状态 (2026-06-23)**:  
- **修复结果**: ✅ 已修复。
- **修复逻辑**: 后端库存计算公式已统一修正为 `累计确认到货量 - 累计施工使用量 - 累计施工损耗量`。并在 `list_arrival_aggregates` 汇总逻辑以及负库存强拦截计算逻辑中彻底贯彻“到货确认后全面入账”原则。非 `cancelled` 且 `arrived_confirm_at` 不为空的确认到货量均立即划归为可用库存（若单据尚未被施工接收，则临时以“确认到货量”计入库存进行退避保护）。

### 2. 数据库迁移失败处理不当

**问题描述**:  
应用启动时执行的数据库迁移如果失败，只打印错误但不阻止应用启动，可能导致应用在不完整的数据库状态下运行。

**代码位置** (`workspace.py:81-121`):
```python
def run_db_migration():
    session = SessionLocal()
    try:
        session.execute(text("ALTER TABLE tube.tube_daily_usage ADD COLUMN IF NOT EXISTS loss_qty ..."))
        session.commit()
    except Exception as e:
        session.rollback()
        print(f"DB Migration for loss_qty and operation_logs failed: {e}")  # ❌ 只打印错误
    finally:
        session.close()

run_db_migration()  # 模块加载时运行
```

**影响**:
- 如果 `loss_qty` 字段创建失败，后续保存损耗量时会出现数据库错误
- 如果审计日志表创建失败，所有审计操作都会失败
- 错误只打印到控制台，可能被忽略

**修复方案**:
```python
def run_db_migration():
    session = SessionLocal()
    try:
        session.execute(...)
        session.commit()
    except Exception as e:
        session.rollback()
        logger.error(f"数据库迁移失败: {e}")
        raise RuntimeError("数据库迁移失败，应用无法启动") from e  # ✅ 抛出异常阻止启动
    finally:
        session.close()
```

**更好的方案**:  
使用专业的数据库迁移工具（如 Alembic），而不是在应用启动时执行 DDL 语句。

**优先级**: 🔴 **P0 - 立即修复**  
**预计工作量**: 2小时  
**风险**: 高（可能导致数据不一致）

**修复状态 (2026-06-23)**:  
- **修复结果**: ✅ 已修复。
- **修复逻辑**: 重构了 `run_db_migration()` 的异常捕获机制。当数据库 DDL 自动迁移（包括 `loss_qty` 字段添加与 `operation_logs` 审计日志建表等）发生任何物理 Exception 时，系统将通过日志模块 `logger.error` 详细记录错误，并显式抛出 `RuntimeError`，从而彻底阻断应用的初始化和启动。防止在数据库 Schema 缺失的错误状态下带病运行。

### 3. N+1 查询导致性能问题

**问题描述**:  
在保存实际使用量时，对每条记录都执行 3 次独立的数据库查询，导致严重的性能问题。

**代码位置** (`demand_management_service.py:203-262`):
```python
def save_usage_records(...):
    for item in records:  # ❌ 循环内执行查询
        # 查询1: 累计到货量
        arrived_row = session.execute(sql_arrived, {...}).first()
        total_arrived = ...
        
        # 查询2: 历史使用量
        usage_before_row = session.execute(sql_usage_before, {...}).first()
        total_use_before = ...
        
        # 查询3: 在途量
        pending_row = session.execute(sql_pending, {...}).first()
        pending_arrival = ...
```

**性能影响**:
- 如果一次保存 10 个型号的使用量，会执行 **30 次数据库查询**
- 在高并发场景下会严重影响响应时间
- 可能导致数据库连接池耗尽

**修复方案**:
```python
def save_usage_records(...):
    # 1. 批量查询所有需要的数据
    pipe_model_ids = [item["pipe_model_id"] for item in records]
    
    # 一次查询获取所有型号的到货量

        SELECT pipe_model_id, SUM(...) as total

    # 2. 在内存中进行校验
    for item in records:
        total_arrived = arrived_map.get(item["pipe_model_id"], 0)
        # ...
```

**优先级**: 🔴 **P0 - 尽快修复**  
**预计工作量**: 4小时  
**风险**: 中（影响用户体验但不影响数据准确性）

**修复状态 (2026-06-23)**:  
- **修复结果**: ✅ 已修复。
- **修复逻辑**: 彻底重构了 `save_usage_records` 保存实际使用与损耗的接口逻辑。将原本在 for 循环中多次执行的 SQL 单条匹配查询剥离。重构为：在循环外部，首先收集本次保存的所有型号 `pipe_model_ids`，利用 PostgreSQL 的批量 `ANY` 语法一键式高效拉取对应的累计到货量、历史使用量和在途量。在内存中利用映射 Dictionary 进行高速缓存与防超限硬拦截逻辑匹配，将数据库连接往返交互次数缩减为恒定的 3 次。

### 4. ECharts 图表实例未清理导致内存泄漏

**问题描述**:  
前端 Dashboard 组件创建的 ECharts 图表实例在组件卸载时未销毁，导致内存泄漏。

**代码位置** (`DashboardView.vue`):
```javascript
let barChartInstance = null
let radarChartInstance = null

onMounted(() => {
  barChartInstance = echarts.init(barChartRef.value)
  radarChartInstance = echarts.init(radarChartRef.value)
})

// ❌ 缺少 onBeforeUnmount 钩子
```

**影响**:
- 用户频繁切换页面时，内存占用不断增加
- 可能导致浏览器卡顿或崩溃
- 在移动端设备上影响更明显

**修复方案**:
```javascript
onBeforeUnmount(() => {
  barChartInstance?.dispose()
  radarChartInstance?.dispose()
  barChartInstance = null
  radarChartInstance = null
})
```

**优先级**: 🔴 **P0 - 尽快修复**  
**预计工作量**: 0.5小时  
**风险**: 中（影响用户体验）

**修复状态 (2026-06-23)**:  
- **修复结果**: ⚠️ 误报（无需改动）。
- **修复逻辑**: 经代码深度走读确认，在前端中央大盘视图 `DashboardView.vue` 组件中，原本就已挂载了完备的 `onBeforeUnmount` 生命周期钩子，且在钩子内部正确执行了 `barChartInstance?.dispose()` 与 `radarChartInstance?.dispose()` 等注销逻辑，防止了内存泄漏。本项警告属于静态分析工具未完全解析组件逻辑的 **伪阳性 (False Positive) 误报**，代码已验证为安全合规，无需进行物理修改。
