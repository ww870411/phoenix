## 2. Phoenix平台背景说明

Phoenix当前不是单页面小工具，也不是为tube项目临时搭建的新站点，而是一个已经运行中的平台级Web应用。tube项目应作为Phoenix中的新业务子系统接入。

### 2.1 Phoenix当前整体形态

Phoenix为“统一登录 + 项目选择 + 多项目模块挂载”的平台式应用：

1. 用户先登录平台；
2. 登录后进入项目选择页；
3. 根据账号权限进入不同项目空间；
4. 各项目在同一平台壳层下运行，但拥有独立页面、接口、数据配置与业务逻辑；
5. 平台统一提供认证、权限、路由壳层、公共组件、部署体系与部分通用服务。

### 2.2 前端基本情况

Phoenix前端基于：

1. `Vue 3`
2. `Pinia`
3. `Vue Router`
4. `Vite`

前端根入口：

```text
frontend/src/main.js
```

全局路由入口：

```text
frontend/src/router/index.js
```

项目页面组织方式：

```text
frontend/src/pages/
frontend/src/projects/
```

新tube项目前端目录建议为：

```text
frontend/src/projects/insulation_pipe_supply_2026/
```

### 2.3 后端基本情况

Phoenix后端基于：

1. `FastAPI`
2. 统一 `/api/v1` 路由前缀
3. 项目级路由注册机制

后端根入口：

```text
backend/main.py
```

总路由入口：

```text
backend/api/v1/routes.py
```

项目路由注册表：

```text
backend/api/v1/project_router_registry.py
```

tube项目后端目录建议为：

```text
backend/projects/insulation_pipe_supply_2026/
```

统一API前缀建议为：

```text
/api/v1/projects/insulation_pipe_supply_2026/...
```

### 2.4 数据与配置目录

建议保留以下项目数据目录：

```text
backend_data/projects/insulation_pipe_supply_2026/
```

其中放置：

1. `tube_config.json`
2. 测试数据或初始化数据；
3. 项目相关导入导出文件；
4. 后续必要的文档或运行配置。

## 6. 配置层落地方案

### 6.1 `tube_config.json`建议结构

V5.1确认配置文件中应包含施工单位配置及施工单位与换热站映射关系。

```json
{
  "supply_entities": [],
  "demand_entities": [],
  "pipe_models": [],
  "manager_assignments": [],
  "construction_units": [],
  "construction_assignments": []
}
```

### 6.2 配置项说明

| 配置项 | 内容 | 说明 |
|---|---|---|
| `supply_entities` | 供给主体 / 管厂 | 记录供给单位名称、简称、联系人、联系电话、状态等。 |
| `demand_entities` | 换热站 / 需求主体 | 记录换热站名称、区域、标段、施工状态等。 |
| `pipe_models` | 保温管型号字典 | 记录型号、管径、单位、分类等。 |
| `manager_assignments` | 现场负责人绑定关系 | 记录规划发展部现场负责人与换热站的绑定关系。 |
| `construction_units` | 施工单位配置 | 记录施工单位名称、联系人、账号关联、状态等。 |
| `construction_assignments` | 施工单位与换热站映射关系 | 记录施工单位与需求主体之间的独立对应关系。 |

### 6.3 配置字段预留原则

1. 换热站配置中应预留区域字段。
2. 换热站配置中应预留标段字段。
3. 换热站配置中应预留施工状态字段，如未开工、施工中、已完工。
4. 这些字段第一阶段可以先用于筛选或隐藏备用，不必全部做成复杂页面功能。
5. 后续看板、权限、筛选和复盘需要时，可逐步启用。

### 6.4 映射关系特别说明

1. 一个现场负责人可以绑定多个换热站。
2. 一个施工单位可以绑定多个换热站。
3. 施工单位与换热站的映射关系不必与现场负责人映射关系一致。
4. 现场负责人负责计划、到货确认、实际使用填报。
5. 施工单位负责其绑定换热站的施工接收确认。

## 7. 数据库核心表建设

### 7.1 第一阶段核心业务表

| 序号 | 表名 | 作用 | V5.1处理意见 |
|---|---|---|---|
| 1 | `tube_baseline_quantity` | 设计用量、计划采购量 | 第一阶段建设，作为核心业务基准数据。 |
| 2 | `tube_daily_plan` | 每日、每站、每型号计划量 | 第一阶段建设，长表保存，不只保存未来三天。 |
| 3 | `tube_delivery` | 发货、到货、施工接收、库管确认生命周期主表 | 第一阶段建设，作为物流链状态主表。 |
| 4 | `tube_daily_usage` | 每日、每站、每型号实际使用量 | 第一阶段建设，长表保存。 |
| 5 | `tube_inventory_adjustment` | 库存调整、盘点差异、退库、调剂、破损、纠错 | 第一阶段建设表结构，页面弱化。 |

### 7.2 不单独建设的表

第一阶段不建设以下独立表：

1. `tube_user_permission`：沿用Phoenix平台账户和权限文件；
2. `tube_delivery_confirmation`：合并进入`tube_delivery`生命周期主表；
3. `tube_exception_record`：第一阶段不做复杂异常闭环；
4. 库存余额快照表：第一阶段实时计算库存余额，不双写库存。

### 7.3 `tube_delivery`状态机

V5.1确认状态如下：

```text
已发货待到货
    ├─ 撤销 → 已撤销（仅限到货确认前）
    ↓
已到货待接收
    ↓
已接收待库管确认
    ↓
已完成
```

异常可通过字段标记：

```text
abnormal_flag = true
remark / arrived_remark / received_remark / warehouse_remark 记录原因
```

## 8. 权限与角色约束

### 8.1 角色清单

| 角色 | 查看范围 | 操作范围 |
|---|---|---|
| `global_admin` | 全部数据 | 配置维护、基准数据调整、特殊系统管理。 |
| `site_manager` | 绑定换热站数据 | 未来三日计划、到货确认、实际使用填报。 |
| `construction_unit` | 映射换热站相关接收事项 | 施工接收确认。 |
| `supplier` | 全部需求、库存、缺口 | 仅操作本供给主体发货记录。 |
| `warehouse_keeper` | 库管台账相关流水 | 仅库管确认。 |
| `viewer` | 授权范围内看板或台账 | 只查看。 |

### 8.2 库管人员限制

V5.1确认库管人员第一阶段：

1. 可以查看库管台账；
2. 可以执行库管确认；
3. 不可以修改发货记录；
4. 不可以修改到货确认；
5. 不可以修改施工接收；
6. 不可以修改未来三日计划；
7. 不可以修改实际使用量；
8. 不可以修改基准数据；
9. “不同意”或争议处理留到第二阶段。

### 8.3 施工单位限制

1. 施工单位通过`construction_assignments`确定换热站范围。
2. 施工单位只确认其映射范围内换热站的接收事项。
3. 施工单位不得确认到货。
4. 施工单位不得填报计划。
5. 施工单位不得填报实际使用量。
6. 施工单位不得操作库管确认。

## 9. 撤销、回退与历史修改规则

### 9.1 发货撤销规则

V5.1确认：

1. 发货记录仅在“已发货待到货”状态下允许撤销。
2. 撤销不等于删除，必须保留原始发货记录。
3. 撤销时必须记录撤销人、撤销时间和撤销原因。
4. 撤销后状态应变更为“已撤销”。
5. 一旦进入“已到货待接收”及后续状态，第一阶段禁止撤销。

### 9.2 确认后禁止回退

V5.1确认第一阶段禁止以下反向回退：

1. 到货确认后的回退；
2. 施工接收确认后的回退；
3. 库管确认后的回退；
4. 已完成状态回退到前置状态。

如发生异常，第一阶段通过备注、异常标记、库存调整预留表或线下处理；正式异常闭环留到第二阶段。

### 9.3 历史计划和历史使用记录不允许修改

V5.1确认：

1. 历史计划记录第一阶段不允许修改。
2. 历史实际使用记录第一阶段不允许修改。
3. 相关纠错机制留到第二阶段设计。
4. 第一阶段如发现问题，可先通过备注、异常说明、线下核实方式处理。

## 10. 库存与缺口计算规则

### 10.1 库存实时计算

第一阶段不建设库存余额表，库存按业务数据实时计算：

```text
当前现场库存 = 累计施工接收量 - 累计实际使用量 ± 库存调整量
```

### 10.2 未来三日缺口

```text
未来三日计划量 = 当日计划量 + 次日计划量 + 第三日计划量

现场硬缺口 = max(未来三日计划量 - 当前现场库存量, 0)

净待发缺口 = max(未来三日计划量 - 当前现场库存量 - 已发货待到货量 - 已到货待接收量, 0)
```

V5.1确认：净待发缺口默认扣减“已到货待接收量”。但该数量仍需单独展示，不能混同为已入账库存。

## 11. 前端页面开发顺序

V5.1确认页面开发顺序如下：

```text
1. demand_management
2. supply_management
3. warehouse_management
4. dashboard
```

### 11.1 第一优先：`demand_management`

原因：需求侧是业务数据源头。

首版能力：

1. 未来三日计划填报；
2. 到货确认；
3. 每日实际使用填报；
4. 本换热站库存与缺口查看；
5. 现场负责人“我的换热站”默认过滤视图。

### 11.2 第二优先：`supply_management`

首版能力：

1. 需求汇总；
2. 供给侧缺口查看；
3. 发货登记；
4. 本供给主体发货进度追踪；
5. “已发货待到货”状态下发货撤销；
6. 超时未确认提醒。

### 11.3 第三优先：`warehouse_management`

首版能力：

1. 库管台账；
2. 状态筛选；
3. 库管确认；
4. 手续留痕备注；
5. 只看、只确认，不做其他修改。

### 11.4 第四优先：`dashboard`

V5.1确认dashboard定位：

1. 汇总优先；
2. 图形便利；
3. 警示为辅；
4. 只查看，不操作。

首版能力：

1. 全局指标卡；
2. 简洁图形展示；
3. 需求侧聚合台账；
4. 供给侧聚合台账；
5. 库管状态聚合台账；
6. 风险预警辅助区；
7. 可跳转明细，但不直接执行发货、确认、修改等操作。
