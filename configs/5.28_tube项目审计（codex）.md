# tube 项目当前逻辑审计报告

日期：2026-05-28  
项目：`insulation_pipe_supply_2026`  
范围：后端 API/service、前端 Dashboard/需求/供应/库管页面、跨端业务口径一致性  
审计性质：只读审计，未修改业务代码  

## 1. 执行摘要

当前 tube 项目的核心业务链路已经形成完整雏形：需求计划、供应发货、到货接收、库管入库、库存与缺口看板均已具备可运行结构。但审计发现，项目中存在一类需要优先治理的系统性问题：同一业务含义被多个逻辑共同控制，且这些逻辑分散在后端接口、后端服务、前端页面计算和前端本地字典中。

这类问题短期内可能表现为“页面数字不一致”“按钮可点但状态不允许”“看板解释与明细不一致”；长期会导致状态机失真、库存口径漂移、领导看板与业务页面相互矛盾。

建议优先修复顺序：

1. 收口全局管理员物流编辑入口，禁止绕过状态机写入不一致数据。
2. 后端集中库存、净缺口、硬缺口计算，前端只展示后台结果。
3. 统一物流状态字典、状态动作、角色权限与页面样式来源。
4. 拆分“实际到货量”和“可用入账量”，避免 KPI 与库存解释混淆。

## 4. 详细发现

### F-001 全局管理员编辑入口绕过物流状态机

严重度：高  
类型：状态机一致性风险、数据污染风险  

#### 现象

标准物流流转入口分别对到货、接收、入库进行了状态限制和数量校验：

- 到货确认只允许 `pending_arrival`。
- 接收确认只允许 `pending_receive`。
- 入库确认只允许 `pending_warehouse`。

但全局管理员编辑入口 `super_update_delivery_record` 可以直接改状态、数量、异常标记、取消字段等核心字段，没有复用同一套状态推进校验。

#### 影响

可能产生以下不一致数据：

- `received_qty > arrived_qty`
- `arrived_qty > shipped_qty`
- 状态已进入 `completed`，但缺少到货/接收/入库确认时间
- 非法状态值进入数据库
- 异常状态与实际数量不一致
- 已取消记录仍携带异常业务数量

这些数据会继续影响供应汇总、库存、OTD、DOI、UCR、SSR 和物流明细。

#### 建议

- 将全局管理员编辑入口改为调用统一状态机校验函数。
- 若必须支持“超级修正”，也应增加状态不变量校验。
- 状态、数量、时间戳应作为一个整体事务校验，不允许只改其中一部分。

#### 验收标准

- 任意入口都不能写入 `received_qty > arrived_qty`。
- 任意入口都不能写入 `arrived_qty > shipped_qty`。
- `completed` 必须具备完整确认时间链路，或显式标记为管理员历史修正。
- 非法状态值被后端拒绝。

### F-002 库存与缺口口径分散且存在负库存处理差异

严重度：中高  
类型：同一业务多处计算、页面口径漂移  

#### 现象

供应汇总中，站点库存按 `total_arrived_qty - total_usage_qty` 计算，可能出现负数。需求矩阵中，同类库存概念又被 `max(..., 0)` 归零。Dashboard 继续基于这些字段派生展示指标。

#### 影响

同一站点可能出现：

- 供应页面显示负库存。
- 需求页面显示零库存。
- Dashboard 硬缺口或安全供应度基于另一套逻辑重新计算。

这会直接削弱看板可信度。

#### 建议

- 在后端服务层新增统一计算函数，例如 `compute_station_inventory`、`compute_net_gap`。
- 明确业务规则：负库存是展示为真实异常，还是统一归零并单独输出异常字段。
- 所有页面统一消费后端返回结果，不再各自计算。

#### 验收标准

- 同一站点、同一日期在供应、需求、Dashboard 中库存口径一致。
- 净缺口只扣减 `pending_arrival`。
- 负库存策略有明确字段和解释。

### F-003 硬缺口与 Dashboard KPI 由前端重复推导

严重度：中  
类型：指标口径漂移、前端承担业务计算  

#### 现象

Dashboard 负责计算 OTD、DOI、PCR、UCR、SSR 等指标，同时在前端多处计算硬缺口。硬缺口不是后端统一返回的业务字段。

#### 影响

一旦后端净缺口、库存或未来计划口径调整，Dashboard 可能仍沿用旧逻辑，造成：

- KPI 卡片与业务明细不一致。
- 指标弹窗解释与后台真实规则不一致。
- 雷达图与卡片指标来自不同推导路径。

#### 建议

- 后端统一返回 `hard_gap_qty`、KPI 原始分子/分母、计算结果和解释文本。
- 前端仅负责格式化、图表展示和弹窗说明。
- Dashboard 不再本地自造业务结果。

#### 验收标准

- Dashboard KPI 可追溯到后端返回字段。
- 前端无重复硬缺口计算。
- KPI 弹窗展示的公式与后端字段一一对应。

### F-004 物流状态字典、标签、动作规则多处维护

严重度：中  
类型：状态字典漂移、权限动作错配  

#### 现象

物流状态在后端接口、供应页面、库管页面、需求页面中均有不同程度的本地维护。标签、样式、可操作状态和筛选条件分散在多个文件中。

#### 影响

容易出现：

- 同一状态在不同页面显示不同文案。
- 页面按钮允许点击，但后端拒绝。
- 后端新增状态后，部分页面漏更新。
- 已完成/已取消记录在某些页面无法追踪。

#### 建议

- 建立 tube 项目统一状态字典。
- 后端返回状态标签、状态顺序、可执行动作、角色权限。
- 前端抽取共享状态工具，不再每个页面硬编码。

#### 验收标准

- 所有页面同一状态显示一致。
- 状态动作由后端权限和状态机共同决定。
- 新增状态只需改一处核心配置。

### F-005 `total_arrived_qty` 字段名与实际计算语义混杂

严重度：中  
类型：字段语义歧义、指标解释风险  

#### 现象

到货聚合字段命名为 `total_arrived_qty`，但计算中会优先使用 `received_qty`，其次 `arrived_qty`，再兜底 `shipped_qty`。这使字段看起来是“总到货量”，实际更接近“可用入账量或业务兜底量”。

#### 影响

会影响：

- 库存解释。
- UCR 使用消耗率解释。
- 到货与接收损耗的业务追踪。
- 领导看板中“到货量”含义。

#### 建议

拆分字段：

- `total_arrived_qty`：实际到货确认量。
- `total_received_qty`：施工接收确认量。
- `stock_input_qty`：用于库存计算的可用入账量。

#### 验收标准

- 字段名与计算语义一致。
- 库存使用 `stock_input_qty` 或明确业务字段。
- KPI 说明不再混用“到货量”和“可用量”。

### F-006 日期截断规则分散且隐式

严重度：中低  
类型：时间口径不透明  

#### 现象

到货聚合和使用量聚合都使用 `< show_date` 这类截止规则。该规则可能是“截至前一日”的正确业务口径，但目前没有在 API 响应中显式说明。

#### 影响

用户看到 `show_date` 时，可能误以为包含当天数据；但实际当天确认的到货/使用可能未计入。

#### 建议

- 后端统一输出 `stock_cutoff_date` 或 `included_until_date`。
- 页面文案明确“截至某日 24:00 前”或“截至前一日”。

#### 验收标准

- 所有库存、消耗、到货指标都能解释是否包含当天。
- 前端展示与后端截止日期一致。

### F-007 API 返回结构不统一，前端易误用

严重度：中低  
类型：接口契约风险  

#### 现象

部分接口返回 `{ ok, rows }`，部分前端代码曾按数组直接使用，导致 OTD 等指标取数异常。这类问题已经修过一次，但接口契约层面仍值得治理。

#### 影响

后续新增页面或指标时，容易再次误用响应结构，表现为：

- 指标为 0。
- 弹窗无数据。
- 图表空白。
- 页面静默降级到错误结果。

#### 建议

- API wrapper 层统一规范返回结构。
- 对关键接口增加前端类型或运行时断言。
- Dashboard 指标计算前校验字段来源。

#### 验收标准

- 同类列表接口统一返回标准结构。
- 前端页面不直接猜测接口形态。
- 无数据、接口异常、字段缺失能明确提示。

## 5. “多个逻辑作用于同一业务”的专项清单

| 业务对象 | 当前多处逻辑 | 主要隐患 | 收口方向 |
| --- | --- | --- | --- |
| 物流状态机 | 标准确认接口、全局管理员编辑、前端按钮状态 | 绕过状态机、非法状态、按钮错配 | 后端统一状态机与动作元数据 |
| 站点库存 | 供应汇总、需求矩阵、Dashboard 展示 | 负库存处理不一致 | 后端统一库存函数 |
| 净缺口 | 供应汇总、需求矩阵、前端说明 | 扣减对象可能再次漂移 | 后端统一 `net_gap_qty` |
| 硬缺口 | Dashboard KPI、Dashboard 表格聚合 | 前端重复计算 | 后端返回 `hard_gap_qty` |
| KPI 指标 | Dashboard 卡片、雷达图、弹窗解释 | 公式与数据来源可能不一致 | 后端返回指标元数据 |
| 状态标签 | 后端 options、供应页、库管页、需求页 | 文案和动作漂移 | 共享状态字典 |
| 日期截止 | 到货聚合、使用量聚合、页面文案 | 用户误解是否含当天 | API 输出截止日期 |

## 6. 建议修复计划

### P0：状态机与数据不变量

- 为 `super_update_delivery_record` 增加状态、数量、时间戳校验。
- 禁止非法状态值。
- 对取消、完成、异常记录建立明确不变量。

### P1：库存与缺口计算收口

- 后端新增统一库存/缺口计算函数。
- 明确负库存策略。
- 净缺口继续坚持只扣减 `pending_arrival`。
- 前端移除重复业务计算。

### P2：状态字典与动作元数据统一

- 后端输出统一状态字典。
- 前端页面共享状态工具。
- 按角色输出可执行动作，避免页面猜测权限。

### P3：KPI 与字段语义治理

- 后端输出 Dashboard KPI 结果、分子、分母和公式说明。
- 拆分 `total_arrived_qty`、`total_received_qty`、`stock_input_qty`。
- 统一 Dashboard 卡片、雷达图、弹窗解释的数据来源。
